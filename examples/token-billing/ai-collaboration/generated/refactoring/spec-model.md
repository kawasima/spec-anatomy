### 上位概念A: 「枠内請求」と「超過込み請求」の二分化

**現状の問題**

ステップ3の `請求` は単一の型で、`超過トークン = 0` の場合と `超過トークン > 0` の場合を区別していません。これは典型的な「整数 0 で業務状態を兼用する」設計です。

要件のAC3とAC4を見ると:

- AC3（枠内）: 30,000 from quota, 0 overage, **$0.00 charge**
- AC4（超過込み）: 20,000 from quota, 30,000 overage, **$0.60 charge**

枠内請求は「無償の利用記録」であり、超過込み請求は「課金が発生した請求」です。**会計的な意味**（売上計上の有無）が異なります。これは整数フラグ（`超過トークン > 0`）で済ませてよい違いではなく、型として分けるべき業務概念です。

**Refactoring**

```text
data 請求 = 枠内請求 OR 超過込み請求

data 枠内請求 = 請求ID
            AND 顧客ID
            AND プロンプトトークン数 AND 完了トークン数
            AND 総トークン数
            AND 枠消費トークン   // この型では: 枠消費トークン = 総トークン数
            AND 計算日時
            AND 適用料金プランID
// Why: 枠内に収まった利用は売上0であり、
//      「課金額」「超過レート」「超過トークン」を持たない方が業務的に正直。
//      ゼロを書く列を増やすと、後続の集計や帳票で「ゼロは超過なし」を毎回判定する必要が生じる。

data 超過込み請求 = 請求ID
              AND 顧客ID
              AND プロンプトトークン数 AND 完了トークン数
              AND 総トークン数
              AND 枠消費トークン   // 0 以上、当月残枠と等しい（使い切り）
              AND 超過トークン     // 1 以上（型としてゼロを許さない）
              AND 適用超過レート   // 計算時点のスナップショット
              AND 請求金額         // > 0
              AND 計算日時
              AND 適用料金プランID
// Why: 超過込みの請求は売上計上対象であり、
//      「どのレートで」「いくら課金したか」が後から再現できなければならない。
//      適用超過レートをスナップショットとして請求側に持つことで、
//      料金プラン側のレート変更に対して請求記録が独立に意味を保つ。
//      （source-material.md の「適用超過レートの請求記録への記録」と整合）
```

これにより `0 円の請求` と `課金額のある請求` は別の型になり、コンパイル段階・仕様レベルで取り違えを防げます。

### 上位概念B: 「当月の枠消費状況」を一段抽象化する

**現状の問題**

ステップ3では `当月使用済みトークン` と `月間込み枠` の差分を毎回 max(0, ...) で計算していました。`当月残枠` という生の整数を持ち回ると、利用パターンの判定（「すでに枠を使い切っているか」）が呼び出し側に分散します。

実際、請求を計算する behavior は「今回の申告が枠内に収まるか／超過するか／そもそも枠が残っていないか」の3パターンで分岐します。これは整数比較ではなく業務状態として上位概念にすべきです。

**Refactoring**

```text
data 当月の枠消費状況 = 枠未消費 OR 枠一部消費 OR 枠使い切り

data 枠未消費   = 月間込み枠トークン                    // 当月使用済み = 0
data 枠一部消費 = 月間込み枠トークン AND 当月使用済みトークン  // 0 < 使用済み < 月間込み枠
data 枠使い切り = 月間込み枠トークン AND 当月超過済みトークン  // 使用済み >= 月間込み枠
// Why: 申告が「枠内に収まる」「枠を跨ぐ」「全量超過になる」の判定は、
//      本質的にこの3状態に対する分岐である。
//      整数比較で書くと、0境界・等号・残量計算のオフバイワンが
//      呼び出し側に拡散する。状態を型で表すと、後段の「請求を計算する」が
//      パターンマッチで全域に書け、漏れが構造的に防げる。
```

### 上位概念C: 「申告と枠消費状況の合成」が請求の種類を決める

**Refactoring**

`請求を計算する` の入力を、上の `当月の枠消費状況` を使って書き直します。

```text

    検証済み使用申告
    AND 当月の枠消費状況
    AND 適用料金プラン     // 超過レート（スナップショット用）
    AND 計算時点
    -> 枠内請求 OR 超過込み請求

// 場合分け（仕様レベル）:

// - 枠未消費 かつ 申告 > 月間込み枠          -> 超過込み請求（枠分=月間込み枠、超過分=申告-月間込み枠）

// - 枠一部消費 かつ 申告 > 残枠              -> 超過込み請求（枠分=残枠、超過分=申告-残枠）

// Why: 「枠を先に消費する」規則を1か所に閉じ込め、

//      呼び出し側は「結果が枠内だったか」をフィールド検査ではなく型で判定できる。
```

### 上位概念D: 「適用料金プラン」と「料金プラン」の区別

**現状の問題**

ステップ3では `料金プラン` と `顧客サブスクリプション` を別々に持って、請求計算の入口で2つを引き回していました。請求を計算する behavior にとって本当に必要なのは「この計算時点で、この顧客に適用される料金プランの内容」だけです。サブスクリプション ID や有効期間は計算には使いません（スタンプ結合）。

**Refactoring**

//      下流の請求計算は「すでに確定したプラン内容」だけを受け取る。
```

data 検証済み使用申告  = 検証済み顧客ID AND プロンプトトークン数 AND 完了トークン数

// ===== 当月の枠消費状況（上位概念B） =====

data 当月の枠消費状況 = 枠未消費 OR 枠一部消費 OR 枠使い切り

data 枠未消費   = 月間込み枠トークン
data 枠一部消費 = 月間込み枠トークン AND 当月使用済みトークン   // 0 < 使用済み < 月間込み枠
data 枠使い切り = 月間込み枠トークン AND 当月超過済みトークン    // 使用済み >= 月間込み枠

data 当月使用済みトークン  = 1以上の整数
data 当月超過済みトークン  = 月間込み枠以上の整数

// ===== 請求（上位概念A） =====

data 請求 = 枠内請求 OR 超過込み請求

data 枠内請求 = 請求ID
            AND 検証済み顧客ID
            AND 適用料金プランID
            AND プロンプトトークン数 AND 完了トークン数
            AND 総トークン数
            AND 枠消費トークン       // = 総トークン数
            AND 計算日時

data 超過込み請求 = 請求ID
                AND 検証済み顧客ID
                AND 適用料金プランID
                AND プロンプトトークン数 AND 完了トークン数
                AND 総トークン数
                AND 枠消費トークン        // 0以上
                AND 超過トークン          // 1以上（型保証）
                AND 適用超過レート        // 料金プランからのスナップショット
                AND 請求金額              // 0より大きい（型保証）
                AND 計算日時

data 適用料金プランID = 料金プランID
data 適用超過レート   = 超過レート

// ===== エラー型（イミュータブルな失敗事実） =====

data 顧客未登録エラー         = 顧客ID
data トークン非負違反エラー   = 違反フィールド AND 与えられた値
data サブスクリプション無効エラー = 検証済み顧客ID AND 計算時点

data 違反フィールド = プロンプトトークン OR 完了トークン

// ===== 振る舞い =====

behavior 顧客IDを検証する = 顧客ID -> 検証済み顧客ID OR 顧客未登録エラー
// Why: 未登録顧客への請求は売掛先が定まらず、会計整合性を欠く（AC1）。

behavior トークン数を検証する = 申告生入力 -> 検証済み使用申告 OR トークン非負違反エラー
// Why: 負のトークン数は物理的にあり得ず、後段の計算結果を毀損する（AC2）。

behavior 有効な料金プランを取得する =
    検証済み顧客ID AND 計算時点 -> 適用料金プラン OR サブスクリプション無効エラー
// Why: 顧客がいずれのプランにも属さない瞬間は、込み枠も超過レートも決まらない。
//      請求金額の計算根拠を欠くため、ここで失敗として確定させる。

behavior 当月の枠消費状況を集計する =
    検証済み顧客ID AND 計算時点 AND 月間込み枠トークン -> 当月の枠消費状況
// Why: 暦月内の過去請求から使用済みトークンを集計し、3状態のいずれかに分類する。
//      整数の比較ではなく型として状態を返すことで、
//      下流の請求計算の場合分けを構造的に網羅させる。
//      （月境界の定義は元素材に明示なし、推測: UTC暦月）
// 全域性: 履歴ゼロでも「枠未消費」を返せるため、失敗ケースなし。

behavior 請求を計算する =
    検証済み使用申告
    AND 当月の枠消費状況
    AND 適用料金プラン
    AND 計算時点
    -> 枠内請求 OR 超過込み請求
// Why: 「込み枠を先に消費する」規則を1か所に閉じ込める。
//      返り値の型は業務状態（枠内 / 超過込み）と1対1。
//      入力はすべて検証済み・確定済みの型に絞られているため、失敗ケースを持たない（全域性）。

workflow 使用申告から請求を生成する =
    申告生入力
    -> トークン数を検証する         // 純粋検証は最初に
    -> 顧客IDを検証する
    -> 有効な料金プランを取得する
    -> 当月の枠消費状況を集計する
    -> 請求を計算する
    -> 枠内請求 OR 超過込み請求

// ===== 不変条件（property としての記述） =====

// 共通
//   forall 請求.
//     総トークン数 = プロンプトトークン数 + 完了トークン数
//
// 枠内請求
//   forall b: 枠内請求.
//     枠消費トークン(b) = 総トークン数(b)
//     b には 超過トークン・適用超過レート・請求金額 のフィールドが存在しない（型保証）
//
// 超過込み請求
//   forall b: 超過込み請求.
//     総トークン数(b) = 枠消費トークン(b) + 超過トークン(b)
//     超過トークン(b) >= 1
//     請求金額(b) = round_half_up( (超過トークン(b) / 1000) * 適用超過レート(b), 2 )
//     請求金額(b) > 0
//   // Why: 通貨丸めは round_half_up を採用（推測。元素材は精度のみ規定）
```

# ステップ7: 6条件の自己検証

最終形が SMDD の6条件をどう満たしているかを箇条書きで自己評価します。

1. **仕様レベルで抽象概念を作る（手続き抽象・データ抽象）**
   - データ抽象として `当月の枠消費状況`（枠未消費 / 枠一部消費 / 枠使い切り）と `請求`（枠内請求 / 超過込み請求）の2つの上位概念を導入した。
   - 手続き抽象として `当月の枠消費状況を集計する` と `請求を計算する` を分け、それぞれ単一の業務責務を持つ。
   - `適用料金プラン` という射影型を導入し、計算に必要な情報だけを下流に渡す。

2. **データの不変条件が一意に定まる（状態ごとに型を分ける）**
   - 「請求」を `枠内請求 OR 超過込み請求` に分割。`枠内請求` は超過関連フィールドを持たず、`超過込み請求` は `超過トークン >= 1` `請求金額 > 0` を型として保証する。
   - 「当月の枠消費状況」も3状態に分けたため、各状態で意味を持つフィールドだけを持つ。`枠使い切り` 状態で「残枠」を引き回さない。
   - 検証済み型（検証済み顧客ID、検証済み使用申告）を導入し、未検証の値が下流に流れないようにした。

3. **振る舞いが全域性を満たす**
   - 失敗の可能性がある behavior は出力型のORに失敗ケースを含める（顧客未登録エラー、トークン非負違反エラー、サブスクリプション無効エラー）。例外を投げない。
   - `当月の枠消費状況を集計する` は履歴ゼロでも「枠未消費」を返せるため失敗を持たない。
   - `請求を計算する` は入力がすべて確定済みの型に絞られているため失敗を持たない（純粋計算）。

4. **振る舞いがスタンプ結合でない**
   - `請求を計算する` は `顧客サブスクリプション` 全体ではなく `適用料金プラン` という射影型だけを受け取る。サブスクID・有効期間は計算で使わない。
   - `当月の枠消費状況を集計する` は顧客全体ではなく `検証済み顧客ID` と `計算時点` だけを受け取る。
   - エラー型は失敗の特定に必要な情報だけを持ち、入力全体を抱えない。

5. **振る舞いが単一責務（Honest & Complete な命名）**
   - 検証（`顧客IDを検証する` `トークン数を検証する`）／参照（`有効な料金プランを取得する`）／集計（`当月の枠消費状況を集計する`）／計算（`請求を計算する`）が分離されている。
   - 命名は何をするかと何を返しうるかが一致している（Honest）。請求を計算するは「請求のいずれかの種別」を返すと宣言済み（Complete）。

6. **各ルールに「Why」が添えられている**
   - 検証ルール（顧客存在・トークン非負）には会計整合性・物理的妥当性の Why を添えた。
   - 「込み枠を先に消費する」規則は README 補足からの引用とともに、顧客への約束の意味を Why として明示した。
   - 「適用超過レートを請求にスナップショット」は将来のレート変更からの独立性として Why を書いた。
   - 推測した部分（暦月の境界、通貨単位、丸め規則）は「推測:」と明示した。

# ステップ8: 仕様テスト性質

最終形のモデルから、property-based test で検証できる性質を `forall ...` の形で列挙します。

```text
// ----- 構造的不変条件（任意の請求に対して） -----

forall b: 請求.
  総トークン数(b) = プロンプトトークン数(b) + 完了トークン数(b)

forall b: 枠内請求.
  枠消費トークン(b) = 総トークン数(b)

forall b: 超過込み請求.
  総トークン数(b) = 枠消費トークン(b) + 超過トークン(b)
  AND 超過トークン(b) >= 1
  AND 請求金額(b) > 0

// ----- 計算規則（請求を計算する の出力に対して） -----

forall (申告, 状況, プラン, 時点) where
    申告: 検証済み使用申告
    AND 状況: 当月の枠消費状況
    AND プラン: 適用料金プラン
    AND 時点: 計算時点.
  let b = 請求を計算する(申告, 状況, プラン, 時点) in
    // 「枠を先に消費する」規則
    枠消費トークン(b) = min( 総トークン数(申告), 残枠(状況) )
    AND
    超過トークン候補 = max( 0, 総トークン数(申告) - 残枠(状況) )
    AND
    ( 超過トークン候補 = 0  =>  b: 枠内請求 )
    AND
    ( 超過トークン候補 > 0  =>  b: 超過込み請求
                              AND 超過トークン(b) = 超過トークン候補
                              AND 請求金額(b) = round_half_up( 超過トークン(b) / 1000 * 超過レート(プラン), 2 ) )
// ※ 残枠は状況の型に応じて: 枠未消費 -> 月間込み枠 / 枠一部消費 -> 月間込み枠 - 当月使用済み / 枠使い切り -> 0

// ----- 全域性（失敗ケースの網羅） -----

forall raw: 申告生入力.
  let r1 = トークン数を検証する(raw) in
    r1: 検証済み使用申告 OR r1: トークン非負違反エラー

forall id: 顧客ID.
  let r2 = 顧客IDを検証する(id) in
    r2: 検証済み顧客ID OR r2: 顧客未登録エラー

forall (cid, t).
    cid: 検証済み顧客ID AND t: 計算時点.
  let r3 = 有効な料金プランを取得する(cid, t) in
    r3: 適用料金プラン OR r3: サブスクリプション無効エラー

// ----- 単調性・境界性質 -----

// 同月内で、より多くのトークンを使うほど枠消費は単調に増えるか枠を超え、
// 越えた瞬間以降は超過が単調に増える
forall 申告A, 申告B where 同顧客 AND 同月 AND 総トークン数(A) <= 総トークン数(B).
  let bA = 連続適用された請求(A), bB = 連続適用された請求(B) in
    枠消費トークン(bA) <= 枠消費トークン(bB)
    AND ( bA: 超過込み請求  =>  bB: 超過込み請求 )
    AND ( bA: 超過込み請求 AND bB: 超過込み請求
            =>  超過トークン(bA) <= 超過トークン(bB) )

// ----- 再現性（適用超過レートのスナップショット） -----

forall b: 超過込み請求.
  let プラン_計算時点 = 計算時点における料金プラン(b) in
    適用超過レート(b) = 超過レート(プラン_計算時点)
// Why: 計算時点での料金プランのレートと、請求に保存された適用超過レートが一致する。
//      その後にプラン側のレートが変わっても請求は影響を受けない。

// ----- 非マイナス性（型では捕えきれないが念のため） -----

forall b: 請求.
  プロンプトトークン数(b) >= 0
  AND 完了トークン数(b) >= 0
  AND 枠消費トークン(b) >= 0
  AND ( b: 超過込み請求  =>  超過トークン(b) >= 1 AND 請求金額(b) > 0 )
```

これらの property は、Core レイヤの仕様テストとして機能します。Shell（API・永続）に依存しないため、仕様モデル単体で実行可能なテストとして書けます。
