# トークン課金 全域性テスト

[仕様モデル](../spec-model/token-billing.md) の `behavior` が、入力区分のすべてに対して定義された出力型に着地することを確認します。元の要件文書の AC1〜AC5 を含み、それを超えて Core の型分割から導かれる入力区分まで網羅します。

## 対象 behavior

```text
behavior 請求を計算する =
  当月消費 AND 料金プラン AND 申告トークン -> 枠内請求 OR 超過込み請求
```

## 入力区分の列挙

`合計 = 申告.プロンプト + 申告.補完`、`枠残量 = max(料金プラン.月間含み枠 - 当月消費.これまでの合計, 0)` とします。区分は次の組合せで定義されます。

| 区分 | 当月消費  | 月間含み枠 | 申告合計 | 枠残量 | 期待される出力             |
| ---- | --------- | ---------- | -------- | ------ | -------------------------- |
| T-01 | 0         | 100,000    | 0        | 100,000 | 枠内請求（消費なし）       |
| T-02 | 0         | 100,000    | 30,000   | 100,000 | 枠内請求                   |
| T-03 | 60,000    | 100,000    | 30,000   | 40,000  | 枠内請求                   |
| T-04 | 70,000    | 100,000    | 30,000   | 30,000  | 枠内請求（境界・ぴったり）|
| T-05 | 80,000    | 100,000    | 50,000   | 20,000  | 超過込み請求（超過 30,000）|
| T-06 | 100,000   | 100,000    | 1        | 0       | 超過込み請求（超過 1）     |
| T-07 | 150,000   | 100,000    | 50,000   | 0       | 超過込み請求（超過 50,000）|
| T-08 | 0         | 0          | 1,000    | 0       | 超過込み請求（無料枠なしプラン）|

T-04 と T-06 は境界条件で、`合計 ≤ 枠残量` の `≤` の側にきちんと枠内請求が落ちることを確認します。

## 期待値の具体例（AC4 由来）

T-05 のケースは元の AC4 に対応します。

```text
入力:
  当月消費.これまでの合計 = 80,000
  料金プラン.月間含み枠 = 100,000
  料金プラン.超過レート = 0.0200 USD/1K
  申告.プロンプト + 申告.補完 = 50,000

計算:
  枠残量 = 100,000 - 80,000 = 20,000
  合計 50,000 > 枠残量 20,000 → 超過込み請求
  枠内消費 = 20,000
  超過 = 50,000 - 20,000 = 30,000
  請求額 = (30,000 / 1000) × 0.0200 = 0.60 USD

期待出力:
  超過込み請求 {
    枠内消費トークン数 = 20,000
    超過トークン数 = 30,000
    適用超過レート = 0.0200
    請求額 = 0.60
  }
```

T-03 のケースは AC3 に対応します。

```text
入力:
  当月消費.これまでの合計 = 60,000
  料金プラン.月間含み枠 = 100,000
  申告合計 = 30,000

期待出力:
  枠内請求 { 枠内消費トークン数 = 30,000 }
  → 超過 0、請求額 0.00（型の上では存在しない）
```

## 全域性が欠けたら検出されるテスト

仕様の漏れを検出するための「ありえない返答」を明示的にテストにします。

- 枠内請求が返るはずの T-04 で、超過込み請求が返ってきたら不合格
- 超過込み請求が返るはずの T-06 で、枠内請求が返ってきたら不合格
- いずれの区分でも例外が投げられたら不合格（仕様の外に漏れている）
- 計算結果の `合計トークン数(申告)` と `枠内消費 + 超過` が一致しないなら不合格（不変条件違反、[invariants.md](invariants.md) と連動）

## API レイヤーの全域性

API ハンドラ（[../shell/api/token-billing-api.md](../shell/api/token-billing-api.md)）の入力区分も含めて全域性を取ります。

| ケース | 入力                                                | 期待ステータス | 期待 `kind`            |
| ------ | --------------------------------------------------- | -------------- | ---------------------- |
| AC1    | customerId が存在しない                             | 404            | `NotFound`             |
| AC2-1  | promptTokens = -1                                   | 400            | `ValidationError`      |
| AC2-2  | completionTokens = -5                               | 400            | `ValidationError`      |
| AC3    | T-03 の入力                                         | 201            | `InQuota`              |
| AC4    | T-05 の入力                                         | 201            | `WithOverage`          |
| AC5    | 任意の正常系                                        | 201            | レスポンスに必須フィールド全部 |
| 追加   | 顧客は存在するがサブスクリプションがない             | 409            | `InvariantViolation`   |
| 追加   | promptTokens, completionTokens の両方が 0           | 201            | `InQuota`（T-01 相当） |

## property-based test として書ける性質

具体例だけでなく、性質として検証できる項目:

```text
∀ (当月消費, 料金プラン, 申告トークン).
  let 結果 = 請求を計算する(当月消費, 料金プラン, 申告トークン) in
    結果.枠内消費トークン数 + 結果.超過トークン数 = 申告.プロンプト + 申告.補完
    AND 結果.枠内消費トークン数 ≤ 料金プラン.月間含み枠
    AND (結果が枠内請求 ⇔ 結果.超過トークン数 = 0)
    AND (結果が超過込み請求 ⇒ 結果.請求額 ≥ 0)
```
