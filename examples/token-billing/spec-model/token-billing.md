# トークン課金

LLM API のトークン使用量から、月間枠と超過レートに基づいて請求額を計算するサービスの仕様モデルです。

## 設計の意図

このモデルは、SMDD の6条件（抽象概念・不変条件・全域性・スタンプ結合排除・単一責務・Why）を満たすように作られています。

- 請求結果を「枠内請求」と「超過込み請求」の2つに OR で分け、超過レートが意味を持つのは超過込み請求のみであることを型で固定する
- 「当月消費」「料金プラン」を別の概念として独立させ、課金計算がそれらをどう参照するかを behavior の入力で明示する
- トークン数・金額・月といった値の不変条件（非負・特定の通貨単位・暦月）を data 定義レベルで宣言する
- 計算ロジックの中に Why を残す（なぜ1000で割るのか、なぜ枠を先に消費するのか）

## 基本概念

```text
data 顧客ID = 文字列 // 空文字列ではない
data 顧客 = 顧客ID AND 顧客名

data トークン数 = 整数 // 0以上
data 金額 = 小数 // 0以上、USD 2桁精度

data 月 = 年 AND 月番号 // 1〜12、暦月単位
```

トークン数と金額は別の値オブジェクトとして分けます。トークン数は整数、金額は USD で 2 桁精度（小数第3位以下は計算過程で丸める）です。

## 料金プランとサブスクリプション

```text
data プランID = 文字列
data 料金プラン =
  プランID
  AND プラン名
  AND 月間含み枠      // トークン数（0以上）
  AND 超過レート       // 1000 トークンあたりの USD（0以上）

data サブスクリプションID = UUID
data サブスクリプション =
  サブスクリプションID
  AND 顧客ID
  AND プランID
  AND 開始日
  AND 終了日           // null 許容、null は無期限
```

サブスクリプションは「顧客がいつからいつまでどのプランに加入していたか」を表します。ある月の請求計算には、その月の最初の日（または使用申告日）に有効なサブスクリプションを参照します。

## トークン使用申告

```text
data 申告トークン =
  プロンプトトークン数
  AND 補完トークン数

// 派生値
合計トークン数(申告) = プロンプトトークン数 + 補完トークン数
```

申告は「この回の API 利用で消費したトークン数」を表します。プロンプトと補完を分けて受け取りますが、課金計算上は合計トークン数のみが意味を持ちます。両方とも 0 以上です。

## 当月消費

```text
data 当月消費 =
  顧客ID
  AND 月
  AND これまでの合計トークン数  // 当月の API 利用すべての合計
```

当月消費は、これから申告される回を**含まない**、それまでの当月分の累積です。請求計算は「当月消費 + 申告トークン」で当月の含み枠を消費していくことになります。

## 請求結果（型分割）

請求結果は2つの状態に分かれます。当月の累積消費 + 今回の申告 が含み枠に収まるか超過するかで分かれ、超過する場合のみ超過レート適用が意味を持ちます。

```text
data 請求 = 枠内請求 OR 超過込み請求

data 枠内請求 =
  顧客ID
  AND 申告トークン
  AND 合計トークン数(申告)
  AND 枠内消費トークン数      // = 合計トークン数(申告)
  // 超過トークン数 = 0、請求額 = 0 は型から自明

data 超過込み請求 =
  顧客ID
  AND 申告トークン
  AND 合計トークン数(申告)
  AND 枠内消費トークン数      // 0以上、含み枠の残量分
  AND 超過トークン数           // 1以上、合計トークン数 - 枠内消費トークン数
  AND 適用超過レート          // 計算時点のサブスクリプションのプランから取得
  AND 請求額                   // (超過トークン数 / 1000) × 超過レート、USD 2桁精度
```

枠内請求と超過込み請求を別の型として分けることで、「超過レートが何だったか」「請求額がいくらか」が意味を持つのは超過込み請求のときだけだ、ということが型レベルで固定されます。boolean フラグや「超過トークン数 = 0 なら超過レートを参照しない」という暗黙ルールに頼りません。

## 計算式の Why

```text
請求額 = (超過トークン数 / 1000) × 適用超過レート

// なぜ 1000 で割るか:
//   超過レートは「1K（1000）トークンあたりの USD」で表現される（プラン定義の単位）。
//   外部の値付けの単位を変えずに金額に変換するための単純な比例計算。
// なぜ枠を先に消費するか:
//   含み枠は前払いされたコミットメントとみなし、超過レート課金は枠を使い切った後の追加分にだけ適用する。
//   これにより「同じ消費量でも枠の使い方で金額が変わる」ような不公平を生まない。
```

## 課金記録

請求結果はそのまま保存されます。

```text
data 請求ID = UUID
data 請求記録 =
  請求ID
  AND 請求         // 枠内請求 OR 超過込み請求
  AND 計算日時
```

計算日時は永続化時に確定する技術的属性です。Core では「計算が終わった瞬間」を表すドメインイベントの時刻として扱い、Shell（DB の TIMESTAMP）でその実体を持ちます。

## behavior 定義

```text
behavior 当月消費を取得する =
  顧客ID AND 月 -> 当月消費
// 当月の累積トークン消費量を返す。当月分の請求記録がなければ 0。

behavior 有効な料金プランを取得する =
  顧客ID AND 日付 -> 料金プラン OR サブスクリプションなし
// 指定日に有効なサブスクリプション（開始日 ≤ 日付 ≤ 終了日 または 終了日 null）を引き、
// そのプランIDから料金プランを返す。
// 該当するサブスクリプションがなければ「サブスクリプションなし」を返す。

behavior 請求を計算する =
  当月消費 AND 料金プラン AND 申告トークン -> 枠内請求 OR 超過込み請求
// 計算ロジック:
//   合計 = 申告トークンの合計
//   枠残量 = max(料金プラン.月間含み枠 - 当月消費.これまでの合計トークン数, 0)
//   if 合計 ≤ 枠残量 then
//     枠内請求 { 枠内消費トークン数 = 合計 }
//   else
//     超過 = 合計 - 枠残量
//     枠内請求 = 枠残量
//     請求額 = round_half_up((超過 / 1000) × 料金プラン.超過レート, 2)
//     超過込み請求 { 枠内消費トークン数 = 枠残量, 超過トークン数 = 超過, 適用超過レート, 請求額 }

behavior 請求記録を保存する =
  請求 AND 計算日時 -> 請求記録
// 計算結果に計算日時を付与して請求記録にする。永続化は Shell が担う。
```

## 6条件のチェック

このモデルが6条件をどう満たしているか:

1. **仕様レベルで抽象概念を作る**: 「枠内請求」「超過込み請求」「当月消費」「適用超過レート」が型として表現されている
2. **データの不変条件が一意に定まる**: 枠内請求には超過トークン数や請求額が存在しない。超過込み請求には超過トークン数 ≥ 1 と適用超過レートが必須。トークン数・金額は非負
3. **振る舞いが全域性を満たす**: `請求を計算する` は合計と枠残量の大小関係で必ず2つの結果型のいずれかに着地する
4. **振る舞いがスタンプ結合でない**: `請求を計算する` は顧客集約や請求記録全体ではなく、計算に必要な3つの値（当月消費・料金プラン・申告トークン）だけを受け取る
5. **振る舞いが単一責務である**: 計算と保存と取得が別々の behavior に分かれている
6. **各ルールに Why が添えられている**: 1000 で割る理由、枠を先に消費する理由、型分割の理由をコメントで明記

## ありえない遷移・ありえない組合せ

型レベルで防がれるもの:

- 枠内請求に対して「超過レートはいくらだったか」を聞く → そのフィールドが存在しない
- 超過込み請求で超過トークン数 = 0 → 型定義により最低 1 以上
- サブスクリプションのない顧客に対して `請求を計算する` を呼ぶ → 入力型に料金プランが必要なため、`有効な料金プランを取得する` が「サブスクリプションなし」を返した時点で計算には進めない
- 顧客が存在しない状態で `当月消費を取得する` を呼ぶ → 顧客の存在確認は Shell の入力境界（API ハンドラ）で先に行う

## このモデルが扱わないこと（Scope Out）

元の要件文書に従い、以下は扱いません:

- 顧客の作成・更新・削除（CRUD）
- 過去の請求記録の検索・取得
- 月初の含み枠リセットの実装方法（Core では「当月消費」が暦月で区切られていることだけを前提とし、リセットは Shell の月境界処理に委ねる）
