# 出張申請

出張申請システムの仕様モデルの最終形です。

## 設計の意図

このモデルは、SMDD の6条件（抽象概念・不変条件・全域性・スタンプ結合排除・単一責務・Why）を満たすように作られています。

- 状態ごとに data を別の型として分け、状態に応じて必須となる項目を type レベルで保証
- 「事前承認要件」という業務概念を導入し、なぜ事前承認が必要かを型として表現
- behavior は入力型を絞り、不適切な状態への呼び出しを型で防ぐ
- 各判定規則に Why コメントを添える
- 承認者を誰にするかの決定（上長、居なければ部門長）を独立した behavior と Why として明示する。before ではこの分岐は提出イベントの処理詳細に if で埋没するが、after では `承認者を決定する` という名前と理由を持つ仕様になる
- 下書き・取消・差し戻し後の再申請といった、業務が端から端まで回るための状態遷移も型と behavior で表現する

## 基本概念

```text
data 部門 = 部門コード AND 部門名 AND 部門長   // 部門長は承認者になりうる社員
data 社員 = 社員番号 AND 氏名 AND 役職 AND 所属部門   // 所属部門は部門への参照
data 出張者 = 社員
data 承認者 = 社員

data 費目 = 交通費 OR 宿泊費 OR 交際費
data 費用負担区分 = 自社負担 OR 先方負担
```

社員に所属部門を、組織に部門を持たせるのは、承認者の決定（後述）と経理連携で参照する所属部門コードの出所を仕様の中に確保するためです。部門は必ず部門長を持ちます。この不変条件があるおかげで「承認者を決定する」が全域になります（誰が申請しても最低でも所属部門の部門長という承認者に着地する）。

## 出張予定

```text
data 出張予定日程 = 出張予定日付 AND 費目 AND 予定金額
data 出張予定費用 = List<出張予定日程> // 1件以上

data 出張予定 =
  出張目的
  AND 出張期間
  AND List<出張者>     // 1件以上
  AND 出張予定費用
  AND 費用負担区分
```

## 出張申請の状態

申請は時間軸の中で次の状態を取ります。状態ごとに data を別の型として分けることで、各状態で必須な項目を型レベルで保証します。

```text
data 出張申請 = 出張申請ドラフト OR 申請済み出張申請 OR 取消済出張申請

data 出張申請ドラフト = 出張予定 AND 申請者
data 申請済み出張申請 = 出張予定 AND 申請者 AND 申請日時
data 取消済出張申請 = 申請済み出張申請 AND 取消日時
```

出張申請ドラフトは提出前の下書きです。何度でも保存・編集でき、提出せずに破棄もできます。ドラフトには申請日時がありません。提出という事実がまだ発生していないからで、申請日時を持つのは申請済み出張申請以降だけです。

取消済出張申請は、いったん提出された申請を最終承認前に取り下げた終端状態です。取消日時という「取り下げが発生した事実」を1つ持ちます。取消済に対しては入力型に取消済出張申請を取る behavior が存在しないため、そこから先へは進めません（終端であることを型で表現）。

申請済み出張申請は、事前承認が必要か不要かによってさらに分類されます。

```text
data 事前承認必要な出張申請 = 申請済み出張申請 AND 事前承認要件
data 事前承認不要な出張申請 = 申請済み出張申請
```

## 事前承認要件（なぜ必要かを型として表現）

```text
data 事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請

// 理由:
// - 高額出張: 高額支出は組織として事前に妥当性を確認する必要がある
// - 役職なし申請: 役職なしの社員は支出判断の権限を持たないため上長の判断が必要
// - 先方負担申請: 接待との境界が曖昧になるため不適切な支出を防ぐ事前チェックが必要
```

事前承認要件を boolean ではなく型として刻むことで、なぜ事前承認が必要かが業務概念として残ります。条件式の羅列に落ちません。

## 事前承認の結果

```text
data 事前承認OK = 申請済み出張申請 AND 事前承認者 AND 承認日時
data 事前承認NG = 申請済み出張申請 AND 事前承認者 AND 承認日時 AND 却下理由
```

## 出張実績と最終承認

出張完了後の状態です。

```text
data 出張実績日程 = 出張実績日付 AND 費目 AND 実績金額
data 出張実績費用 = List<出張実績日程> // 1件以上
data 出張実績 = 出張予定 AND 出張実績費用

data 最終承認 = 出張実績 AND 最終承認者 AND 最終承認日時
```

## behavior 定義

```text
behavior 下書きを保存する =
  出張予定 AND 申請者 -> 出張申請ドラフト
// 新規作成・更新の両方を兼ねる。何度でも呼べる（提出前のドラフトは何度でも上書きできる）
// Why: 下書きは業務上の確定した事実ではなく作業中の状態なので、日時属性を持たせず上書き可能にする

behavior 下書きを破棄する =
  出張申請ドラフト -> Unit
// 提出前の下書きを捨てる。Why: 未提出の下書きには「取り下げるべき申請」という事実がまだ無いので、
// 取消（取消済出張申請を残す）ではなく破棄（痕跡を残さない削除）とする

behavior 承認者を決定する =
  申請者 -> 承認者
// Why: 承認は指揮命令系統の上位者が行う。申請者に上長が設定されていればその上長を、
//       上長が居なければ申請者の所属部門の部門長を承認者とする。
//       部門は必ず部門長を持つ（部門 = 部門コード AND 部門名 AND 部門長）ため、
//       上長が未設定の社員でも承認者は必ず1人に定まり、承認の空白が生じない（全域性）。

behavior 出張申請する =
  出張申請ドラフト AND 申請日時 -> 申請済み出張申請
// ドラフトを提出して申請済みにする。申請日時という「提出が発生した事実」がここで加わる

behavior 事前承認が必要か判断する =
  申請済み出張申請 -> 事前承認必要な出張申請 OR 事前承認不要な出張申請
// 判定ロジック:
// - 出張予定費用の合計が 100,000円 以上 → 高額出張
// - 申請者の役職が「役職なし」 → 役職なし申請
// - 費用負担区分が「先方負担」 → 先方負担申請
// 上記のいずれかに該当すれば「事前承認必要」、いずれにも該当しなければ「事前承認不要」

behavior 上長が事前承認する =
  事前承認必要な出張申請 AND 承認者 AND 承認日時 -> 事前承認OK OR 事前承認NG
// 前提条件: 承認者が申請者の上長であること
// 結果: 事前承認OKなら出張実施可能。事前承認NGなら却下理由を記録

behavior 差し戻された申請を再申請する =
  事前承認NG AND 出張予定 AND 申請日時 -> 申請済み出張申請
// 却下（差し戻し）された申請を、申請者が出張予定を修正して再提出する。
// Why: 事前承認NGを終端にすると、却下は「やり直しのきかない失敗」になってしまう。
//       修正して申請済みに戻せる経路を型として用意することで、却下は差し戻しとして扱える。
// 全域性: 出力は申請済み出張申請の1枝に必ず着地する（再提出は必ず申請済みへ戻る）

behavior 出張実績を登録する =
  事前承認OK OR 事前承認不要な出張申請 AND 出張実績費用 -> 出張実績
// 出張完了後、実費用を含めて実績を登録する

behavior 最終承認する =
  出張実績 AND 承認者 AND 最終承認日時 -> 最終承認
// 前提条件: 承認者が申請者の上長であること
// 立て替えた金額を経理に連携する

behavior 出張申請を取り消す =
  ( 申請済み出張申請
    OR 事前承認必要な出張申請
    OR 事前承認不要な出張申請
    OR 事前承認OK
    OR 事前承認NG
    OR 出張実績 )
  AND 取消日時 -> 取消済出張申請
// 申請者が提出済みの申請を取り下げる。取り下げが起きた事実として取消日時を記録する。
// Why: 取り消せるのは最終承認より前まで。入力の OR に 最終承認 を含めないことで、
//       「最終承認済み以降は取り消せない」を型レベルで表現する。
//       出張申請ドラフトも入力に含めない。未提出の下書きは取消ではなく破棄で捨てる。
```

## 6条件のチェック

このモデルが6条件をどう満たしているか:

1. **仕様レベルで抽象概念を作る**: 「事前承認要件」「事前承認OK／NG」「出張実績」「取消済出張申請」「部門」のような業務概念が型として表現されている
2. **データの不変条件が一意に定まる**: 状態ごとに type を分けたため、各状態で必須な項目が型レベルで保証される（事前承認OK には承認者と承認日時が必須、取消済出張申請には取消日時が必須、部門には部門長が必須）
3. **振る舞いが全域性を満たす**: 各 behavior は入力型に応じて必ず出力型のいずれかに着地する。`承認者を決定する` は部門が必ず部門長を持つため（上長が居なくても部門長へ）必ず承認者に着地し、`差し戻された申請を再申請する` は必ず申請済み出張申請へ戻る
4. **振る舞いがスタンプ結合でない**: behavior は必要な data だけを受け取る（出張申請全体を渡さない）。`承認者を決定する` は申請者だけ、`出張申請を取り消す` は取り消し対象の状態と取消日時だけを受け取る
5. **振る舞いが単一責務である**: 「事前承認が必要か判断する」「承認者を決定する」「下書きを破棄する」「出張申請を取り消す」のように Honest & Complete な命名
6. **各ルールに Why が添えられている**: 事前承認要件の各バリアントに加え、承認者の決定順（上長→部門長）、取消可能範囲（最終承認前まで）、破棄と取消の使い分けに業務上の理由をコメントで添えている

## ありえない遷移

禁止遷移として、次は型レベルで防がれます。

- 出張申請ドラフトに対して `上長が事前承認する` を呼ぶ → 入力型が一致しない
- 事前承認不要な出張申請に対して `上長が事前承認する` を呼ぶ → 入力型が `事前承認必要な出張申請` のみ
- 申請済み出張申請に対して直接 `最終承認する` を呼ぶ → 入力型が `出張実績` のみ
- 最終承認（およびそれ以降）に対して `出張申請を取り消す` を呼ぶ → 取消の入力 OR に `最終承認` を含めていないため型が一致しない（最終承認前までしか取り消せない）
- 出張申請ドラフトに対して `出張申請を取り消す` を呼ぶ → 取消の入力 OR にドラフトを含めていない（未提出の下書きは `下書きを破棄する` で捨てる）
- 取消済出張申請に対して任意の behavior を呼ぶ → 取消済出張申請を入力型に取る behavior が存在しない（終端）

一方で、事前承認NG（差し戻し）は終端ではありません。`差し戻された申請を再申請する` の入力型に含めているので、申請者が出張予定を修正して申請済み出張申請へ戻せます。却下を「やり直せる差し戻し」として型で表現しています。
