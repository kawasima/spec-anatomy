# 出張申請システム 設計書一式（演習用）

「生成AI時代のドキュメンテーション」ハンズオンの演習素材です。従来のSIerが書いた出張申請システムの設計書一式（Excel）と画面デザインHTML、変換の雛形が入っています。洗練後の仕様モデル（答え）は入っていません。自分の手で作って、確かめて、直すのがこの演習です。

## 用意するもの

- Claude Code（すぐ使えるようにログインまで済ませておく）
- VS Code などのエディタ
- Java 25 以上（最後にコード生成まで試す場合だけ）

## この演習でやること

Excelの設計書をAIに読ませて `requirements.md`（要求のMarkdown）に変換します。そこで終わりにせず、変換した `requirements.md` だけを別のAIに読ませて、「元の設計書を読んだ人にしか答えられない質問」をぶつけます。答えられない箇所が、変換で失われたものです。それを仕分けて、`requirements.md` を直します。「AIに変換させたら完成」ではない、というのを手を動かして知るのが目的です。

## 中身

- `traditional-design/` — 従来SIの設計書一式（Excel）。画面5取引・バッチ・共通コンポーネント・テーブル定義・コード設計・外部インタフェース定義など。出張の申請から、事前承認、実績登録、最終承認、経理連携、取消まで、ひと通り入っています
- `design-html/` — 画面のデザインHTML。Excel側の画面レイアウトは空欄なので、見た目はこちらで補います
- `requirements.template.md` — 変換の雛形。これに沿わせて `requirements.md` を作ります

---

## Step1: 業務ルールを読んで、理由が書いてあるか確かめる

いきなりAIに渡さず、まず自分の目で設計書を読みます。といっても、たくさんあるExcelの中から業務ルールを自分で探し出すのは大変です。そこで、主な業務ルールの「場所」と「概要」はこちらで用意しました。あなたは、その場所を開いて中身を読み、「業務ルール（詳細）」と「その理由が書いてあるか？」を埋めてください。

|場所|業務ルール（概要）|業務ルール（詳細）|理由が書いてあるか？|
|---|---|---|---|
|BTS01 2.6.2 (3)／共通コンポーネント CC0001|事前承認が必要かどうかを判定する|予定費用の合計が10万円以上、または申請者が役職なし、または費用負担が先方負担なら事前承認待ち（'20'）。どれにも当てはまらなければ事前承認不要（'21'）。|書いていない（なぜ10万円か、なぜ役職なし・先方負担が対象かは無い）|
|BTS01 2.6.2 (4)|事前承認の承認者を決める|||
|BTS02 2.6.2／2.6.3|上長が承認、または却下する|||
|BTS03 2.6|出張後に実績費用を登録する|||
|BTS04 2.6.2 (4)／共通コンポーネント CC0002|会社が立て替える精算額を計算する|||
|BTS05 2.6.5|提出済みの申請を取り消す|||
|BTB01 2.6／BTS04 2.6.2 (5)|最終承認済の精算額を経理システムへ連携する|||

1行目は記入例です。場所を開いて、そこに書いてあるルールを「詳細」に自分の言葉で書き、いちばん右の「理由が書いてあるか？」を正直に埋めてください。全部を完璧に書く必要はありません。いくつか埋めてみるだけで十分です。

埋めていくと、たいていの行で「理由が書いてあるか？」が「書いていない」になるはずです。ルールそのもの（10万円、0円、取り消せる状態）は書いてあっても、「なぜそう決めたのか」はどこにも書かれていない。この気づきが、あとで効いてきます。

どこを見ればいいか分からなくなったら、AIに「この設計書のどこに判定条件や状態の変化が書いてあるか、場所だけ挙げて」と聞いて、当たりをつけてから自分で読んでください。ただし「なぜそうするのか」はAIに埋めさせないこと。

## Step2: requirements.md を作る

`requirements.template.md` を作業フォルダに置いて、これに沿って `requirements.md` を作ります。雛形には各要求に「根拠 / Why」と「由来 / Traceability」の欄があります。根拠の欄が埋まらないことで、「設計書に理由が書かれていない」ことが目に見えるようになっています。

受け入れ基準は EARS という書き方に沿わせます。むずかしくありません。次の型でルールを書くだけです。

- WHEN（何かが起きたとき）… THE SYSTEM SHALL（システムはこうする）
- IF … THEN（異常や特定の条件のとき）
- WHILE（ある状態が続く間）
- WHERE（あるオプションが有効なとき）
- 条件がなく常にそうなら、そのまま THE SYSTEM SHALL

例:「WHEN 予定費用が10万円以上の出張申請が登録された THE SYSTEM SHALL 事前承認を必須とする」。この型で書けるのはルールの結果までです。「なぜ10万円なのか」はこの型に収まりません。だから根拠 / Why の欄が別にあります。

雛形を置いたら、Claude Code に次のように頼みます。

```text
requirements.template.md の構成に厳密に沿って、配布された出張申請システムの設計書一式（システム機能設計書(画面) BTS01〜BTS05、システム機能設計書(バッチ) BTB01、共通コンポーネント設計書、テーブル定義書、コード設計書、外部インタフェース設計書）から、同じディレクトリに requirements.md を1本にまとめて生成してください。登録・事前承認・実績登録・最終承認・経理連携・取消を取引横断で Requirement として並べてください。
各 Requirement に User Story、EARS形式の Acceptance Criteria、根拠/Why、由来/Traceability を必ず設けてください。
根拠が元設計書に書かれていない場合は、推測で埋めず 【要業務確認】 とだけ記してください。
最後に「未確定・要業務確認」欄に、あなたが推測で補った点と、設計書に書かれていなかった点をすべて挙げてください。
```

「根拠が書かれていなければ、推測で埋めず 【要業務確認】 と書く」がこの演習の肝です。AIは理由が分からなくても、それらしい理由を作って埋めてしまいます。そうなると欠落が見えなくなります。だから、分からないものは分からないと残させます。

Excelの読み取りに時間がかかったり、表が崩れたりすることがあります。その場合は `MarkItDown`（Microsoftのツール）で設計書をいったんMarkdownに変換してから、雛形に流し込むと楽です。

出来上がった `requirements.md` は、たいてい読むぶんにはきれいに完成して見えます。ここで「できた」と思わないのが、この演習です。次で確かめます。

## Step3: 変換したものだけをAIに読ませて確かめる

`requirements.md` だけを空のフォルダにコピーして、そこで新しい Claude Code のセッションを開きます。これは大事な手順です。設計書のExcelと同じフォルダのまま聞くと、AIはExcelを読みに行って正解してしまい、確かめになりません。`requirements.md` 1枚だけを見せてください。

そのAIに、「元の設計書を読んだ人なら答えられるはずの質問」をぶつけます。良い質問は、内容の確認ではなく、判断の理由や背景を聞くものです。

- なぜこの方式・この金額・この区分にしたのか
- この制約はどこから来ているのか
- この状態のあと、次に何ができるのか

「申請状態は何種類ありますか」のような、書いてあることをなぞるだけの質問だと、AIはすらすら答えてしまって、欠落が見えません。「なぜ」を聞いてください。

質問がうまく作れないときは、Step1のメモが使えます。「理由が書いてあるか？」の列に「書いていない」と埋めた行を、そのまま「なぜ？」の質問に変えるだけです。たとえば、こんな質問になります。

- なぜ事前承認は予定費用10万円からなのか
- なぜ先方負担のとき精算額は0円なのか
- 出張申請はいつまで取り消せるのか。なぜ最終承認のあとは取り消せないのか
- 事前承認が却下されたら、その申請は次に何ができるのか
- 「出張申請」と「出張実績」はなぜ別々なのか

これらをそのまま聞いてもいいですし、自分でも2〜3個、同じ調子で作ってみてください。作ったらAIに聞いて、返ってきた答えを見ます。

答えを聞くときに、いちばん気をつけることがあります。AIが理由をそれらしく答えても、それが元の設計書に書いてあったかを確かめてください。もっともらしい理由でも、AIやあなたが推測で作ったものかもしれません。設計書に無ければ、それは推測です。「AIが答えられた＝OK」ではありません。

## Step4: 振り分けて、requirements.md を直す

このStepは2つに分かれます。まず見つかったものを分類に振り分け（前半）、そのあと振り分けに沿って `requirements.md` を直します（後半）。この直した版が、この演習の成果物「改訂版」になります。

まず振り分けです。Step3で「AIが答えられなかった／推測で答えた」ものや、変換のときに扱いに困ったものを、次の①〜⑦のどれかに振り分けます。分類の中身と、それぞれどうするかは先に決めてあります。あなたは振り分けるだけです。

- ① 書き足せば済む設計の詳細（項目名、検索条件など）→ `requirements.md` の改訂版に書き足す
- ② 図やレイアウトでしか表せないもの（画面イメージなど）→ 本文に書かず、デザインHTMLへのリンクにする
- ③ 別の資料が持っている外部連携の仕様 → 連携仕様書へのリンクにする
- ④ 変更履歴のような運用の記録 → git などに任せる
- ⑤ まだ決まっていない・理由が書かれていないもの → 埋めずに 【要業務確認】 として残す
- ⑥ 設計書どうしで書き方が食い違っているもの → どちらが正か確認する（勝手に決めない）
- ⑦ `requirements.md` に欄が無いカタログ・スキーマ（コード値・メッセージなど）→ `requirements.md` ではない別の置き場（enum・多言語リソース・スキーマ・画面仕様）に置く

次のものを、①〜⑦に振り分けてみてください（1行目は記入例）。

|見つかったもの|振り分け（①〜⑦）|
|---|---|
|なぜ事前承認は10万円からなのか|⑤|
|なぜ先方負担で精算額は0円なのか||
|なぜ最終承認のあとは取り消せないのか||
|事前承認が却下されたあとの動き（差し戻し→再申請）||
|画面の見た目・レイアウト||
|経理連携ファイルの項目レイアウト||
|変更履歴（誰がいつ直したか）||
|経理連携が最終承認画面と日次バッチの両方に書いてある||
|申請状態のコード値10種||
|エラーメッセージの文言||

正解をぴったり当てる必要はありません。なぜそこに振ったかを自分で言えれば十分です（③と⑦のように、どちらでもよいものもあります。どちらも「本文に書かず、別に持つ」で結論は同じです）。

ここからが後半です。振り分けたら、その扱いに沿って `requirements.md` を直します。これがこの演習の成果物「改訂版」です。①は改訂版に書き足します。このとき、設計書の内容をそのまま写すのではなく、「なぜその設計にしたのか」を自分の言葉で推測して書いてみてください。②③はリンクにします。⑦は別の置き場に回します（`requirements.md` には書きません。「欄が無い＝作らない」ではなく、enum や多言語リソースとして別に作り続けます）。雛形の「## この要求フォーマットに載らないもの（別成果物へ）」に、「何を → どの置き場へ」で書き出しておきましょう。

いちばん気をつけるのは⑤です。理由が分からないもの・まだ決まっていないものを、推測で埋めて確定させてはいけません。「まだ決まっていない」という事実を、そのまま 【要業務確認】 として残します。

直し終えたら、改訂版をもう一度読み返して、2つだけ確かめてください。ひとつ、①で書き足したことは設計書に本当に書いてあったか（自分の推測を、事実として書いていないか）。ふたつ、⑤の 【要業務確認】 を、直すうちに埋めて確定させていないか。理由の欄が最後まで 【要業務確認】 のまま残っていても、それで正解です。改訂のゴールは、分からない理由を埋めることではなく、「まだ決まっていない」という事実を正直に残すことだからです。

## この先: お手本を見る

Step4まで終えたら、完成形のお手本を見てみましょう。同じ出張申請を、型と振る舞いと理由（Why）で書き直した仕様モデルが、配布元の spec-anatomy リポジトリの `examples/business-trip/spec-model` と `examples/business-trip/shell` にあります。

自分が `requirements.md` に埋めきれなかったもの、置き場が無かったものが、そこではどこに置かれているかを見比べてください。理由や状態の移り変わりは `spec-model` に、コード値・画面項目・連携ファイルの様式は `shell`（の persistence / api / ui）に分かれて置かれています。

これが唯一の正解というわけでも、これを真似すれば生産性が上がるというわけでもありません。`requirements.md` 1枚に押し込めなかったものが、どこに着地するのかを確かめる、という見方で眺めてください。
