# 従来SI設計書一式（before）

出張申請システムを、従来のSIerが Nablarch開発標準の書式で設計したらどうなるか ── その**本物の設計書一式**です。同じ business-trip の [spec-model](../spec-model/) と [shell](../shell/) が洗練後（after）で、こちらがその手前（before）にあたります。

書式は Nablarch開発標準の実物テンプレート（`_templates/`）に流し込んだもので、自作の模造ではありません。出典とライセンスは [NOTICE.md](NOTICE.md) を参照してください（CC BY-SA 4.0）。

## この一式の使い方

まず設計書を読み、次に [spec-model/business-trip.md](../spec-model/business-trip.md) を読むと、同じ業務が「手続きと状態コード」から「型と behavior」へどう組み替わるかが見えます。組み替えの過程は [ai-collaboration/excel-to-spec-evaluation.md](../ai-collaboration/excel-to-spec-evaluation.md) に記録しています。

この一式は業務を端から端まで回せる範囲を扱います。下書きの保存・編集・破棄、申請提出、事前承認の要否判定と承認・却下、却下後の差し戻し・再申請、実績登録、最終承認、経理連携、そして最終承認前までの取消。after の spec-model / shell も同じ範囲を扱うので、before と after の違いは「機能の抜け」ではなく「設計の構造」だけになります。before が業務として不完全だと、after が良いのか before が忘れていただけか区別できないため、範囲を揃えています。

## 収録している設計書（27ブック）

Nablarch開発標準の設計書体系を、出張申請システムを1サブシステム（サブシステムID=BT）として実体化しています。

- 索引: 設計書一覧
- システム機能設計: システム機能一覧 / システム処理フロー / リクエスト一覧
- 画面設計: 画面一覧 / 画面遷移図 / システム機能設計書(画面) ×5取引（BTS01 出張申請登録 / BTS02 事前承認 / BTS03 出張実績登録 / BTS04 最終承認 / BTS05 出張申請一覧）
- バッチ: システム機能設計書(バッチ) BTB01 経理連携ファイル作成
- 共通コンポーネント設計: 共通コンポーネント一覧 / 共通コンポーネント設計書（CC0001〜CC0004）
- インタフェース設計: 外部インタフェース一覧 / 外部インタフェース設計書 IBT01 経理システム連携ファイル
- 帳票設計: 帳票一覧 / 帳票設計書 RBT01 出張精算書
- メール設計: メール一覧 / メール設計書（MBT01〜MBT03）
- メッセージ設計: メッセージ設計書
- コード設計: コード設計書
- データモデル設計: テーブル一覧 / テーブル定義書 / ドメイン定義書 / 採番一覧
- ジョブフロー設計: ネット・ジョブフロー

## before に残した「従来SIの病」と、after での組み替え

この一式は、リポジトリが批判する設計の癖を消さずに残しています。テーブル駆動や物理項目名むき出しは Nablarch のテンプレート自体が持っている書式であり、わざと作り込んだものではありません。その上に出張申請固有の癖を載せています。各行が after のどの洗練に対応するかを次に示します。

| before（この設計書一式） | after（[spec-model](../spec-model/business-trip.md) / [shell](../shell/)） |
| --- | --- |
| テーブル定義書：BT_APPLICATION 1本に `appl_status`（申請状態コード8〜9値）＋ approver_id / approved_at / rejection_reason / finalized_at を全てNULL許容カラムで持つ | 8状態を型としてOR分割。承認者・承認日時は該当状態にだけ必須 |
| 画面機能設計書(BTS01)：確認画面の登録イベント処理詳細に事前承認要否判定を自然言語のif羅列で埋没（合計≧100000で'20' / 役職コード='99'で'20' / 費用負担='02'で'20' / それ以外'21'）、Why不記載 | `事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請` を型として behavior に。各枝にWhyコメント |
| 共通コンポーネント設計書：CC0002 精算額計算を独立コンポーネントとして設計（自社負担=実費 / 先方負担=0円） | 概念ごと消滅。`最終承認する` behavior の経理連携に吸収された幻の関数 |
| CC0001 事前承認要否判定の判定ロジックが、画面機能設計書(BTS01)と共通コンポーネント設計書に重複（single source of truth なし） | 判定は「事前承認要件」型に一元化。重複が消える |
| 画面機能設計書：画面項目の情報取得元に `BT_APPLICATION.purpose` 等のテーブル.カラム、物理項目名むき出し（テーブル駆動） | 画面項目に業務概念の派生元。予定費用合計は `sum(...)` の純粋関数 |
| 画面引継ぎ項目で出張申請の入力内容一式をまるごと次画面へ引き継ぐ（スタンプ結合） | 状態遷移で必要な型だけを渡す |
| 状態遷移を appl_status のコード値の書き換え（'20'→'30'→'40'→'50'→'60'、取消は'90'）で表現、遷移先分岐もコード値のif羅列 | 状態が型で、behavior の入力型が遷移の前提条件を保証 |
| 最終承認の処理詳細に、経理連携ファイルの固定長レイアウト（桁数・詰め方・0円時のヘッダ/トレーラ出力）が染み出す | 固定長の詳細は Shell の外部I/F（IBT01）に隔離、画面仕様から切り離す |
| 承認者決定（上長→不在なら部門長）を提出イベントの処理詳細に if 羅列で埋没、Why不記載 | `behavior 承認者を決定する : 申請者 -> 承認者` に一元化、上長→部門長fallbackの理由をWhyで明示 |
| 却下（'31'）・下書き（'00'）・取消（'90'）を appl_status の値で持ち回り、差し戻し再申請・取消可否をコード値のif判定で表現 | `事前承認NG`／`出張申請ドラフト`／`取消済出張申請` の型と、`差し戻された申請を再申請する`／`下書きを保存する`／`出張申請を取り消す` の behavior で表現。取消可能範囲・再申請可否を入力型で保証 |

## 図について

画面レイアウト・画面遷移図・システム処理フロー・ネットジョブフロー・帳票レイアウトの図は、生成に用いた openpyxl が Excel の図形を保持できないため、図形キャンバスは空のままにしています。図に相当する内容は after 側の [shell/ui/](../shell/ui/)（画面レイアウトHTML、`screen-transitions.md` の画面遷移）に存在します。

## 再生成

これらの .xlsx は [tools/design-doc-generator/](../../../tools/design-doc-generator/) の生成スクリプトで、空テンプレートに内容を流し込んで作っています。空テンプレート（`_templates/`）はリポジトリに同梱せず（`.gitignore` 済み）、build 時に Nablarch開発標準から取得します。初回はネットワークが必要です。

```sh
cd tools/design-doc-generator
python build.py
```
