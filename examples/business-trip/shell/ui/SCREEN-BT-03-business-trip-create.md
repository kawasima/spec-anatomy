# SCREEN-BT-03 出張申請作成・編集

## この画面の業務的役割

申請者が出張予定（目的・期間・出張者・費用明細・費用負担区分）を入力し、出張申請ドラフトを作成・編集する画面。新規作成モードと編集モードを持ち、編集モードは一覧画面から `出張申請ドラフト`（'00'）行または `事前承認NG`（差し戻し, '31'）行をクリックしてロードする。

3系統の操作を持つ。ドラフト保存は `behavior 下書きを保存する` を呼んで `出張申請ドラフト` を保持する（新規・更新の両方、何度でも呼べる）。破棄は `behavior 下書きを破棄する` を呼んでドラフトを削除する。申請提出は `behavior 出張申請する` を呼んで `申請済み出張申請` に遷移させる。差し戻し（`事前承認NG`）をロードした編集モードでは、申請提出ボタンの代わりに再申請ボタンを出し、`behavior 差し戻された申請を再申請する` を呼んで `申請済み出張申請` に戻す。

## 関連する behavior

- `behavior 出張申請する` ([仕様モデル](../../spec-model/business-trip.md))
- `behavior 下書きを保存する` ([仕様モデル](../../spec-model/business-trip.md))
- `behavior 下書きを破棄する` ([仕様モデル](../../spec-model/business-trip.md))
- `behavior 差し戻された申請を再申請する` ([仕様モデル](../../spec-model/business-trip.md))

この画面が表示・入力対象とする `data`。

- `data 出張予定` ([仕様モデル](../../spec-model/business-trip.md))
- `data 出張申請ドラフト` ([仕様モデル](../../spec-model/business-trip.md))
- `data 事前承認NG` ([仕様モデル](../../spec-model/business-trip.md)) — 差し戻し内容（却下理由）を編集モードで表示

| モード       | 起動元                                          | ロードする状態          | 出す操作ボタン                       |

| 新規作成     | 一覧の「新規申請」ボタン                        | なし（空フォーム）      | 下書き保存 / 申請する / キャンセル   |

- 主アクター: 申請者（全ロール — 自分の申請を作成する）
- 副アクター: なし

### イベント詳細

#### onLoad（編集モードのロード）

- 下書き編集モード: パスパラメータの申請IDで GET /api/v1/business-trips/{id} を呼び、`出張申請ドラフト` をフォームに展開。ドラフト以外がロードされた場合（他ユーザが先に提出した等）は「この申請は編集できません」とトーストし一覧へ戻す
- 差し戻し再申請モード: GET で `事前承認NG` をロードし、出張予定をフォームに展開しつつ差し戻し理由（`却下理由`）を差し戻し理由領域に表示。申請提出ボタンを再申請ボタンに差し替える

#### onSaveDraft

- バリデーション: なし（ドラフトは未入力項目があっても保存可）
- 呼び出す behavior / API: `behavior 下書きを保存する` / PUT /api/v1/business-trips/{id}（新規作成時は POST /api/v1/business-trips）
- 遷移条件:
  - 200 OK → 本画面に留まり「下書き保存しました」とトースト
  - 409 NOT_DRAFT → 「この申請は下書きではないため編集できません」とトーストし一覧へ戻す
  - サーバエラー → エラー画面へ

#### onDiscard

- バリデーション: なし。確認ダイアログ「この下書きを破棄しますか？」で確定した場合のみ実行
- 呼び出す behavior / API: `behavior 下書きを破棄する` / DELETE /api/v1/business-trips/{id}
- 遷移条件:
  - 204 No Content → SCREEN-BT-01 へ遷移し「下書きを破棄しました」とトースト
  - 409 NOT_DRAFT → 「下書き以外は破棄できません」とトーストし一覧へ戻す
  - サーバエラー → エラー画面へ

#### onResubmit

- バリデーション: onSubmit と同じ（必須項目・日付順序・明細1件以上・予定金額1以上）
- 呼び出す behavior / API: `behavior 差し戻された申請を再申請する` / POST /api/v1/business-trips/{id}/resubmit
- 遷移条件:
  - 200 OK → SCREEN-BT-01 へ遷移し「再申請しました」とトースト
  - 400 ValidationError → 本画面に留まりバリデーションエラーを表示
  - 409 NOT_REJECTED → 「差し戻された申請ではないため再申請できません」とトーストし一覧へ戻す
  - 409 NO_APPROVER_FOUND → 「承認者（上長・部門長）が設定されていません」とエラー表示
  - サーバエラー → エラー画面へ

#### onSubmit

- バリデーション: 必須項目（出張目的・出張開始日・出張終了日・費用負担区分・明細1件以上・各明細の全項目）がすべて入力済みであること。終了日 >= 開始日であること。予定金額が1以上であること
- 呼び出す behavior / API: `behavior 出張申請する` / POST /api/v1/business-trips/{id}/submit
- 遷移条件:
  - 200 OK → SCREEN-BT-01 へ遷移し「申請しました」とトースト
  - 400 ValidationError → 本画面に留まりバリデーションエラーを表示
  - 409 NO_APPROVER_FOUND → 「承認者（上長・部門長）が設定されていません」とエラー表示（提出時に事前承認必要と判定され承認者を解決できなかった場合）
  - サーバエラー → エラー画面へ

## 業務的事前条件・事後条件

- 事前条件: ログイン済みであること。下書き編集モードの場合、対象申請が `出張申請ドラフト` 状態であること。差し戻し再申請モードの場合、対象申請が `事前承認NG` 状態であること
- 事後条件:
  - onSaveDraft 完了後は `出張申請ドラフト` 状態を保持
  - onDiscard 完了後はドラフトが削除される（`Unit`）
  - onSubmit / onResubmit 完了後は `申請済み出張申請` 状態へ遷移（`behavior 事前承認が必要か判断する` はサーバ側で連続して呼ばれ、`事前承認必要な出張申請` または `事前承認不要な出張申請` になる。事前承認必要なら `behavior 承認者を決定する` で承認者が解決される）

## 受け入れ基準

| ケース                   | 操作                                          | 期待結果                                                              |
| ------------------------ | --------------------------------------------- | --------------------------------------------------------------------- |
| 正常系（下書き保存）     | 必須項目未入力のまま下書き保存                | ドラフトが保存され「下書き保存しました」トーストが表示される           |
| 正常系（申請提出）       | 全必須項目を入力して申請する                  | 申請済みに遷移し一覧に戻る。「申請しました」トーストが表示される      |
| 業務エラー（バリデーション）| 出張目的を空のまま申請する                  | 「出張目的は必須です」エラーが表示され API を呼ばない                 |
| 業務エラー（日付順序）   | 終了日 < 開始日で申請する                     | 「終了日は開始日以降を指定してください」エラーが表示される            |
| 業務エラー（明細0件）    | 明細を全削除してから申請する                  | 「費用明細を1件以上入力してください」エラーが表示される               |
| 正常系（破棄）           | 下書き編集モードで破棄→確認ダイアログで確定    | ドラフトが削除され一覧へ戻り「下書きを破棄しました」トーストが表示される |
| 正常系（再申請）         | 差し戻し再申請モードで内容を修正して再申請      | 申請済みに戻り一覧へ。「再申請しました」トーストが表示される          |
| 業務エラー（承認者不在） | 上長も部門長も未設定の申請者が事前承認必要な内容で申請する | 「承認者（上長・部門長）が設定されていません」エラーが表示される |
