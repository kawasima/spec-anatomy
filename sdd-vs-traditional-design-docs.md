# Traditional SIer設計書 と 仕様駆動開発の仕様の比較分析

このリポジトリの目的は、Traditional SIerの設計ドキュメントをAI Readableにし、足りないものを足して、SIer向けの仕様駆動開発のSpec Setを作ることである。本文書はその第一段階として、`sdd.md` が想定する仕様駆動開発（Spec-Driven Development／Living Documentation）の世界観と、Traditional SIerの伝統的設計書を突き合わせ、現状を分析する。

設計書サンプルとしては、Nablarch開発標準の35フォーマット体系を `nablarch-design-schema.dsl` で構造化したものを参照する。これは特定のフレームワークを評価するためではなく、Traditional SIerが実務で扱っている設計ドキュメントの典型例として参照しているだけで、本ファイルの観察と結論はNablarch非依存である。

## 突き合わせの基準

`sdd.md` から、仕様駆動が「仕様」に求める性質を抽出すると次のようになる。

| 性質 | 説明 |
| --- | --- |
| Whyを含む | 意図・原則・なぜそうするかが記述されている |
| 機械可読 | DSL／構造化／アノテーション／型として処理可能 |
| コードと冗長 | 同じ事実をコードや実行物にも持ち、テストで同期検査できる |
| AIに渡せる粒度 | 短く局所的、タスクに必要な部分だけ取り出せる |
| 実行可能 | feature／openapi／contract／ArchUnitなど検証ゲートになる |
| 意図決定（人間合意） | 利害・許容範囲・例外を人間が会話で確定する |
| 保守の最短経路 | 仕様を直す方がコードを直すより速い経済が成立する |

これらに照らして、Traditional SI設計書の各要素を A〜E の5区分に振り分ける。区分の意味は次のとおり。

- A: sddが想定する仕様そのもの（Spec-as-source／Spec-anchored に乗る）
- B: 仕様ではなく sensor／harness（テスト・契約検査・規範）
- C: 解釈次第でA/Bになる（Living化可能だが運用次第）
- D: sddが想定しない／コードと冗長で腐るリスクが大きい
- E: sddの世界に対応物がない（リポジトリ／Git／ジェネレータで代替）

このA群は項目・契約・カタログ・参照整合性のレベルにとどまる。次の3つは含まれない。

- 業務概念モデル（集約境界、エンティティのライフサイクル、業務不変条件）。例：「未承認注文 → 承認済み注文」の状態遷移、「在庫を引き当てる」の業務ルール
- 業務不変条件。例：「注文金額 = 明細合計 + 税」「キャンセル済み注文は再開できない」
- ユビキタス言語の意味記述（Why／業務的背景）

これらは Traditional SI の設計書セットには記述する欄がない。システム機能設計書の「処理詳細」が自然言語で書かれるが、これは後述の D 群、つまりコードと冗長で腐るリスクが大きい領域に入る。

## F: 設計書が事前ガイドか事後保守対象か

A〜E とは別の軸で、もうひとつ整理すべき観点がある。設計書が「事前ガイドだけのもの」か「事後も保守されるもの」かという軸だ。仕様駆動が成り立つかどうかは、この軸での位置取りで決まる。

| 性質 | 該当する設計要素 |
| --- | --- |
| Spec-first（事前のみ）で腐りやすい | §3.4 システム機能設計書の処理詳細・編集仕様・処理結果／§4 共通コンポーネント設計書の処理説明・処理概要 |
| Spec-anchored（事後も保守される）に向く | §11 ドメイン定義書（項目バリデーション）／テーブル定義書／§5 外部I/F設計書／§9 メッセージ設計書／§10 コード設計書／§13 単体テスト仕様書 |
| Spec-as-source（コード生成元）に向く | §11 ドメイン定義書（→Bean Validation アノテーション付きFormフィールド）／§11.2 テーブル定義書（→Entity／DDL：既にgsp-dba-mavenが存在）／§5 外部I/F設計書（→Bean Definition）／§9 メッセージ設計書（→i18nリソース）／§10 コード設計書（→enum） |

## A群を仕様駆動として運用している組織はどれだけあるか

A群を改めて見ると、本質的には2つのレイヤーで構成されている。ひとつは項目・契約・カタログ層で、ID体系、項目バリデーション、コード設計、メッセージ、採番、API契約、電文スキーマ、テーブル定義、画面・帳票・メールの一覧がここに入る。もうひとつは参照整合性層で、§18 の不変条件 IC-1〜18（ID参照が壊れていないか、孤児がいないかのチェック）がここに該当する。A群は業務概念モデル（集約・状態遷移・業務不変条件）を含まない。

業界の実態を並べると、おおよそ次のようになる。

- A群の部分を仕様駆動している組織は多い。OpenAPI／Prisma／sqlc／i18n／Zod等を組み合わせる形で、2020年以降の主流になっている
- A群の項目・契約・カタログ全体を構造化フォーマットで揃えている組織は、限定的だが存在する。Java／.NET エンタープライズで Bean Validation＋Flyway＋OpenAPI＋ResourceBundle を組み合わせた構成が、大手SI内製基盤・金融基幹・ERPカスタム開発で採用されている
- Excelで揃えているがLiving Document化していない組織は多い。通常のSI受託案件全般がここに当てはまる
- A群を構造化＋ジェネレータ＋fitness functionまで揃えてSpec-as-source運用している組織は少ないが、珍しいわけではない。スキーマ駆動を全社標準化した Stripe や Shopify、一部の Java エンタープライズが該当する
- A群を揃えたうえで業務概念モデル（集約・状態遷移・業務不変条件）も別途持って仕様駆動している組織は、極めて少ない。F#／Scala／TypeScript with Effect-TS などで関数型ドメインモデリングを徹底するチームが該当する

A群レベルで仕様駆動している組織はそこそこ多い、という認識でよい。Bean Validation＋OpenAPI＋Flyway＋ResourceBundle＋Schema Registry＋ArchUnit という組み合わせは、Java／.NET のモダンなエンタープライズ開発では珍しくない。ただしSI受託案件のように設計書がExcelで管理されている現場では、AI Readableな形式への変換が済んでいないために仕様駆動として機能していない。A群素材を揃えていることと、それがエージェントに渡せる形であることは別問題である。

そしてA群を完全に揃えても、業務概念モデル（集約・状態遷移・業務不変条件・ユビキタス言語）は埋まらない。sdd.md が期待する「人間とAIが同じ業務意図を共有する」レベルに到達するには、業務モデルのDSL（関数型ドメインモデリング、Event Storming成果物等）が別途必要になる。

## 両陣営に欠けている層

ここまではTraditional SI設計書のどの要素がsddのどこに対応するかを並べた対応関係の議論だった。最後に、対応関係の外側、つまりTraditional SI設計書にもsddにも対応物が無い領域に触れておく。

両陣営とも、業務概念モデルを仕様のコアとして置く層を持たない。Traditional SI設計書では業務概念が「処理詳細」の自然言語に紛れ込み、コードとの冗長によって時間とともに古くなる。sdd側では、業務概念モデルを仕様の中心に据える論（Wlaschin系）が選択肢のひとつとして言及されてはいるが、Living Documentation や Harness の議論の中心には組み込まれていない。

ここで言う業務概念モデル仕様は、次の4つを指す。

- 集約境界。トランザクション境界・整合性境界・ライフサイクルを共にする業務概念のまとまり。例：注文集約（注文ヘッダ＋明細）、在庫集約（SKU単位の引当・実在庫）
- ライフサイクル状態遷移。エンティティが取りうる状態と遷移条件、遷移時に発生するイベント。例：「未承認 → 承認済 → 出荷指示済 → 出荷完了 → 完了」と各遷移のトリガ・事後条件
- 業務不変条件。複数項目・複数集約・時間軸をまたいでドメイン上常に成り立つべき制約。例：「注文金額 = 明細合計 + 税」「キャンセル済み注文は再開できない」「在庫引当数 ≤ 実在庫数」
- ユビキタス言語の意味記述。用語の業務的意味と、近接概念との違い。例：「『出荷指示』と『出荷完了』はなぜ別状態か」「『請求』と『売上計上』のタイミング差異」

A群が扱うのは「単一値の妥当性」と「ID参照の整合性」であって、これら4つとはレイヤーが違う。だからA群を完全に揃えても、この層は埋まらない。

Traditional SI設計書には、この4つを書く欄がない。ドメイン定義書は項目バリデーションカタログで、集約境界も状態遷移も業務不変条件も書けない。テーブル定義書はDDLレベルの構造定義で、集約境界はテーブル分割と一致しないことが多く、状態遷移は「ステータスコード列」として痕跡しか残らない。不変条件はID参照整合性チェックだけで、業務不変条件ではない。業務ルールの行き場はシステム機能設計書「処理詳細」の自然言語＋疑似コードになり、これはコードと冗長なのでD群に入る。

sdd側の議論を改めて読むと、業務概念モデルは流派のひとつとして触れられているだけだ。READMEの仕様5流派整理（OpenAPI／BDDシナリオ／ドメインモデル／ADR／スーパープロンプト）では、ドメインモデル中心論は並列の選択肢として置かれている。sdd.md の §1〜§9 は API契約・BDDシナリオ・ADR・型・lint・ArchUnit・実行時ログを軸に組み立てられていて、§5 の Harness 表にも集約・状態遷移・業務不変条件は出てこない。§4 で構造化された仕様・制約・例・テストの必要性に触れてはいるが、業務概念モデルの正本をどこに置くかは扱っていない。sdd.md が描いているのはコード境界（API・型・テスト）に重心がある制御系であり、業務概念モデルを仕様の正本に据える論は別の伝統として認識されているが、骨格には組み込まれていない。

なぜ両者ともこの層を持てなかったかには、それぞれ理由がある。Traditional SI設計書はフレームワークや実装基盤が提供する定型部品の網羅から出発しているので、定型部品では扱えない業務概念モデルは「処理詳細」の自然言語に追いやられる。sdd側はコードと冗長な知識を信頼できる形に保つ仕組み（Martraire）と、エージェント制御系（Böckeler）から出発しているので、コードに直接対応物を持たない業務概念の構造そのものは中心から外れる。一方、業務概念を型・代数的データ構造として直接表現する論（Wlaschin、関数型ドメインモデリング、DDD戦術）は別の伝統として確立しているが、Living Documentation や Harness の議論と統合した文書は存在しない。

3つの伝統が独立に発展した結果、どれかひとつだけを採用しても業務概念モデル仕様は埋まらない。Traditional SI型の網羅的設計書を完璧に揃えても、業務不変条件を書く場所はない。sdd型のharness（API契約＋BDD＋ArchUnit＋ADR）を完璧に揃えても、集約境界の根拠は型シグネチャに痕跡が残るだけで、なぜその境界かは曖昧なままだ。Wlaschin型のドメインモデルを書いても、それがAPI契約や項目バリデーションやBDDシナリオとどう接続するかの規約がない。結果として業務概念モデルはどの陣営の正本にも置かれず、各プロジェクトで属人的にドメインエキスパートの頭の中、処理詳細の自然言語、コードのコメント、ホワイトボードの写真などに分散することになる。sdd.md の問い「この知識は今どこにあるのか？」に対して、「失われているか仮説である」としか答えられない領域がここに残る。

これがSIer向けSpec Setを構築するうえで埋めるべき2つの欠落である。ひとつはA群（項目・契約・カタログ・参照整合性）のAI Readable化、もうひとつは業務概念モデル層の追加。次の段階では、これらを踏まえてSpec Setに含めるべき要素を選別し、Markdownテンプレート群として設計する。

## （補足）Nablarchの「ドメイン定義書」の扱い

最初に位置づけを揃えておきたい設計要素がある。Traditional SI設計書で言う「ドメイン定義書」だ。これは単一入力値（フィールド）に対する型・桁・文字種・コード参照のバリデーション仕様カタログであって、ドメイン名（論理／物理）、データ型、桁数、文字種・数値範囲・コード値・日付書式などのバリデーション処理名、システム許容文字、`@Id` や `@DateFormat("yyyyMMdd")` といった追加アノテーションを並べたものである。

これは Vaughn Vernon や Scott Wlaschin の関数型ドメインモデリング系で言う「ドメインモデル」（業務概念・集約・値オブジェクト・不変条件・ライフサイクル状態）とは別物である。位置づけとしては Java Bean Validation の Constraint カタログに型コード辞書を足したものに近い。本ファイルでは「ドメイン定義書」を業務概念モデルとは扱わず、項目バリデーションカタログ／型辞書として扱う。
