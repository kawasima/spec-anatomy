# workflow 定義の書き方

`workflow` は複数の `behavior` を順序づけて連結したものです。1つの業務プロセス全体を表現します。

このディレクトリには、業務プロセスごとに `workflow` 定義のファイル（例: `submit-business-trip.md`、`fulfill-order.md`）を置きます。

## 基本形

```text
workflow "業務プロセス名" =
  input: 入力データ
  output: 出力データ OR エラーケース

  // step1: ...
  // step2: ...
```

入力と出力は `data` で定義された型を使い、各ステップに連結する `behavior` を記述します。

## 例

```text
workflow "出張申請を提出する" =
  input: 下書き出張申請
  output: 事前承認待ち出張申請 OR 事前承認済み出張申請

  // step1: 事前承認の要否を判定する
  // step2: 判定結果に応じて適切な状態に遷移させる
```

```text
workflow "海外出張申請を受け付ける" =
  input: 未検証出張申請
  output: 承認待ち出張申請 OR エラー

  // step1: 基本的な検証を行う（既存の振る舞いを再利用）
  // step2: 海外出張特有の検証を行う（渡航書類の確認）
  // step3: 外貨予算を確保する
  // step4: 承認ワークフローを開始する
```

## いつ workflow に切り出すか

SMDD のコンポーザビリティの考え方に従えば、新しい業務要件への対応は次の順で考えます。

1. 既存の `behavior` の組み合わせで表現できるか考える
2. 組み合わせで表現できれば、それを `workflow` として記述する
3. 組み合わせで表現できなければ、新しい `behavior` を追加する

`workflow` は「既存の `behavior` を組み合わせた業務プロセス」を表現するためのものであり、`workflow` の中で複雑な業務ルール判定を直接書くことは避けます。判定は個別の `behavior` に切り出します。

## ワークフローの境界

各ステップは1つの `behavior` の呼び出しに対応します。複数の `behavior` を1ステップにまとめると、コンポーザビリティが失われます。

```text
# 良い：各ステップが1つの behavior
workflow "出張申請を提出する" =
  input: 下書き出張申請
  output: 事前承認待ち出張申請 OR 事前承認済み出張申請

  // step1: 事前承認要否を判定する
  // step2: 判定結果に応じて遷移する
```

```text
# 避けたい：1ステップに複数責務
workflow "出張申請を提出する" =
  input: 下書き出張申請
  output: 事前承認待ち出張申請 OR 事前承認済み出張申請

  // step1: 事前承認要否を判定して、判定結果に応じて遷移し、上長に通知する
```

## 副作用は workflow の端に

純粋な変換と副作用を持つ処理を分離します。副作用（メール送信、外部システムへの通知など）は `workflow` の端（最後のステップ）に配置します。途中で副作用が起きると、ワークフローの再実行や巻き戻しが難しくなります。

```text
workflow "事前承認する" =
  input: 事前承認待ち出張申請 AND 承認者
  output: 事前承認済み出張申請

  // step1: 承認者の権限を確認する        （純粋）
  // step2: 承認を記録する                （純粋）
  // step3: 申請者に通知する              （副作用：メール送信）
```

## エラーハンドリング

各ステップで発生しうるエラーは、ワークフロー全体の出力型に `OR` で含めます。各ステップが返す可能性のあるエラーをすべて列挙して、ワークフローの利用者に明示します。

```text
workflow "出張申請を提出する" =
  input: 下書き出張申請
  output: 事前承認待ち出張申請 OR 事前承認済み出張申請 OR 検証エラー OR 在庫不足エラー
```

エラーが累積していくのが煩雑なら、全域性を達成する手段のうち「入力データを制限する」を再検討します。各 `behavior` が安全な入力を要求する形で全域性を達成できれば、`workflow` の出力型もシンプルになります。

## チェックリスト

`workflow` を書いたら次を確認します。

- 入出力の型が `data` で定義されているか
- 各ステップが1つの `behavior` に対応するか
- 副作用がワークフローの端に配置されているか
- 各ステップで発生しうるエラーが出力型に含まれているか
- ワークフローが業務プロセスの全体像を表しているか（中途半端な部分プロセスではない）
