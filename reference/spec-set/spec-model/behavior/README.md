# behavior 定義の書き方

`behavior` は仕様モデルが扱う業務操作を、入力と出力の対応関係として定義します。ドメイン記述ミニ言語の文法と、SMDD の指針（結合度・全域性・behavior のテスト）に従います。

このディレクトリには、業務領域ごとに `behavior` 定義のファイル（例: `order-approval.md`、`business-trip-flow.md`）を置きます。関連する `behavior` は1ファイルにまとめます。

## 基本形

```text
behavior 振る舞い名 = 入力データ -> 出力データ
```

「何を受け取り、何を返すか」だけを書きます。実装の詳細（HTTPやSQL）は登場しません。

```text
behavior 受注する = 注文者 AND 商品 -> 注文
behavior 価格を計算する = 商品 AND 数量 -> 金額
behavior 在庫を確認する = 製品コード -> 在庫数量
```

## 全域性（Totality）

`behavior` は入力 `data` のすべてに対して、結果（成功または失敗）が `data` として定義されている状態を保ちます。例外を投げて済ませる、`null` を返す、未定義の入力を許す、のいずれもしません。失敗も `data` の一枝として明示します。

```text
# 良い：失敗を data として表現
behavior 注文を検証する = 未検証注文 -> 検証済み注文 OR 検証エラー
behavior メールアドレスを検証する = 未検証メールアドレス -> 検証済みメールアドレス OR 検証エラー
behavior 在庫を引き当てる = 製品コード AND 数量 -> 引当済み OR 在庫不足エラー
```

全域性を達成する手段は3つあります。

### 1. デフォルト値を返す

エラーケースに対して業務上意味のあるデフォルト値を返す方法です。「ボランティアの給与は0円」のように業務的に妥当な場合に使います。仕様モデルの段階では、本質的複雑さを隠してしまうリスクがあるため第一選択にはしません。

### 2. 出力データにエラーケースを含める

```text
behavior 整数同士の割り算をする = 被除数 AND 除数 -> 商 OR ゼロ除算エラー
```

エラーの存在を明示できますが、振る舞いを合成するたびにエラーケースが累積する弱点があります。

### 3. 入力データを制限する（推奨）

```text
data ゼロでない整数 = 整数 // 0以外
behavior 整数同士の割り算をする = 被除数 AND ゼロでない整数 -> 商
```

エラーケースをデータ型の境界に押し込むため、`behavior` の内部からエラーが消えます。仕様モデルの段階では、これを第一に検討します。

仕様モデルが Strict Spec としてエージェントに渡されるとき、シグネチャの可読性は決定的に重要です。`割り算する : 被除数 AND ゼロでない整数 -> 商` と書くだけで、この振る舞いの前提条件と保証が読み取れます。

## ガード条件と前提条件

ある `behavior` が適用できるかどうかの条件は、「入力型を絞る」ことで型レベルに刻みます。

```text
behavior 上長が事前承認する =
  事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー
// 前提条件: 承認者IDが申請者の上長IDと一致すること
```

前提条件が型では表現できない場合（複数の集約をまたぐ、外部システムを参照する等）はコメントで明示し、前提条件違反時のエラーを `OR` の一枝として出力に入れます。「契約違反だから何が起きてもよい」とせず、契約違反時の振る舞いも仕様の一部として定義します。

## 状態遷移としての behavior

業務操作によって状態が変わる場合、入力型と出力型がそれぞれ別の状態の `data` になります。

```text
behavior 申請を提出する = 下書き出張申請 -> 事前承認待ち出張申請 OR 事前承認済み出張申請
behavior 事前承認する = 事前承認待ち出張申請 -> 事前承認済み出張申請
behavior 出張を開始する = 事前承認済み出張申請 -> 進行中出張申請
behavior 出張を完了する = 進行中出張申請 -> 完了出張申請
```

`behavior` のシグネチャの集合を見るだけで、どの状態からどの状態へ遷移できるかが分かります。逆に、遷移できないこと（出張完了から下書きへの遷移など）は、`behavior` を書かないことで表現します。

## 依存と副作用

実装では `behavior` は外部の状態を参照したり変更したりします。仕様モデルではこれをコメントで明示します。

### 依存

外部のマスタやカタログを参照する処理。`behavior` の出力は外部の状態によって変わりますが、外部を変更することはありません。

```text
// 依存: 製品カタログ
behavior 製品コードを検証する = 未検証製品コード -> 検証済み製品コード OR 検証エラー

// 依存: 住所マスタ
behavior 住所を検証する = 未検証住所 -> 検証済み住所 OR 検証エラー
```

### 副作用

外部の状態を変更する処理。

```text
// 副作用: メールサーバーにメール送信を依頼する
behavior メールを送信する = 宛先 AND メール本文 -> 送信完了

// 副作用: 在庫管理システムの在庫数を減らす
behavior 在庫を減らす = 製品コード AND 数量 -> 在庫更新済み
```

副作用のある `behavior` はワークフローの端に配置します。純粋な変換と副作用を持つ処理を分離することで、合成の予測可能性が高まります。

## スタンプ結合の排除

`behavior` は使わないデータまで含む大きな型を引数に取らないようにします。実際に使う最小限のデータだけを受け取る型に絞ります。

```text
# 良い：必要な型だけ受け取る
data 出張期間 = 開始日 AND 終了日
behavior 出張期間を検証する = 出張期間 -> 検証済み期間 OR 期間矛盾エラー
```

```text
# 避けたい：使わないフィールドまで受け取る
behavior 出張期間を検証する = 出張申請 -> 検証済み期間 OR 期間矛盾エラー
```

スタンプ結合を排除することは、結合度を下げるだけでなく、`behavior` を別の文脈（出張予約システムなど）でも再利用できる柔軟性に直結します。

## Honest & Complete な命名

SMDD の単一責務と命名の指針です。

- **Honest**: その振る舞いが実際にやっていること以外を名前に含めない
- **Complete**: その振る舞いがやっていることを名前からすべて読み取れる

「申請を処理する」のような名前は Honest でも Complete でもありません。「事前承認が必要か判断する」「出張申請を提出する」「事前承認を却下する」のように、業務上の意味を一意に表す動詞 + 目的語の形にします。

名前に「〜して〜する」（AND）が含まれているのは、複数の責務を持っているサインです。命名プロセス（Naming as a Process）の段階を経て、責務を分解します。

## Why を添える

判定規則には理由を必ず添えます。Why をプロンプトに含めるかどうかでエージェントが生成するコードの形が変わります。

```text
behavior 事前承認が必要か判断する =
  申請済み出張申請 -> 事前承認必要な出張申請 OR 事前承認不要な出張申請

// 事前承認要件:
// - 高額出張(予定費用合計 >= 100,000円)
//   理由: 高額支出は組織として事前に妥当性を確認する必要がある
// - 役職なし(申請者.役職 = なし)
//   理由: 役職なしの社員は支出判断の権限を持たないため上長の判断が必要
// - 先方負担(費用負担区分 = 先方負担)
//   理由: 接待との境界が曖昧になるため不適切な支出を防ぐ事前チェックが必要
```

Why のない `cost >= 100000` のような条件は、エージェントが条件式の羅列として実装します。Why があると、エージェントは `HighCost`、`LackOfAuthority`、`ExternalCostBearing` のような型として要件を表現します。

## エラー型の設計

`behavior` の失敗側に登場するエラー型は、次の3種類に分けて整理します。

### 検証エラー

入力 `data` が不変条件を満たさない場合のエラーです。

```text
behavior 注文金額を検証する = 注文金額 -> 検証済み注文金額 OR 注文金額検証エラー
```

### 権限エラー

操作する主体が、その振る舞いを適用する資格を持たない場合のエラーです。

```text
behavior 上長が事前承認する =
  事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー
// 承認エラーには「承認者IDが上長IDと一致しない」が含まれる
```

### 状態不整合エラー

現在の状態が、その振る舞いの前提状態と合致しない場合のエラーです。

```text
// 「事前承認済み」を入力に要求することで、状態不整合エラーを型レベルで防ぐ
behavior 事前承認する = 事前承認待ち -> 事前承認済み OR 承認エラー
```

エラーを「失敗」と一括りにせず、なぜ失敗したかが仕様から読み取れる粒度を保ちます。

## 振る舞いの連鎖

複雑な業務は複数の `behavior` が順に連なって構成されます。各ステップの入出力をつなぐ形で書きます。

```text
// ステップ1: 注文を検証する
// 依存: 製品カタログ, 住所マスタ
behavior 注文を検証する = 未検証注文 -> 検証済み注文 OR 検証エラー

// ステップ2: 価格を記載する
// 依存: 製品カタログ
behavior 価格を記載する = 検証済み注文 -> 金額記載済み注文

// ステップ3: 在庫を引き当てる
// 依存: 在庫管理システム
behavior 在庫を引き当てる = 金額記載済み注文 -> 引当済み注文 OR 在庫不足エラー

// ステップ4: 受注通知を送る
// 副作用: メール送信
behavior 受注通知を送る = 引当済み注文 -> 受注通知送信済み
```

ステップを連結したものは [workflow/](../workflow/) として記述します。

## チェックリスト

`behavior` を書いたら次を確認します。

- 入力に対して必ず出力が定義されているか（全域性）
- 失敗を例外や `null` ではなく `data` の一枝として表現したか
- 全域性を「入力データの制限」で達成したか（できない場合は理由を明記）
- スタンプ結合になっていないか（使わないデータを引数に取っていないか）
- 名前が Honest & Complete か（「〜して〜する」になっていないか）
- 判定規則に Why が添えられているか
- 依存と副作用がコメントで明示されているか
- エラー型が3種類（検証／権限／状態不整合）のどれかに整理されているか
