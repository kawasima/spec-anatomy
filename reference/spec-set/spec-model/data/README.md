# data 定義の書き方

`data` は仕様モデルが扱う情報の構造を定義します。ドメイン記述ミニ言語の文法と、SMDD の洗練指針（凝集度・結合度・全域性、イミュータブルデータモデル）に従います。

このディレクトリには、業務領域ごとに `data` 定義のファイル（例: `order.md`、`employee.md`、`business-trip.md`）を置きます。1つの `data` ごとにファイルを分けるのではなく、関連する型をまとめて1ファイルにします。

## 単純な値

業務上意味のある最小単位の値は、基本型（文字列・整数・数値・日付）に制約コメントを添えて定義します。

```text
// 2〜100文字
data 名前 = 文字列
// 0以上、円単位
data 金額 = 整数
// "C"で始まる8桁の英数字
data 顧客ID = 文字列
// 0.001以上
data キログラム = 数値
```

Domain Primitive の考え方に従い、プリミティブ型をそのまま使い回さず、業務制約を持つ単純な値として独立させます。

## AND（直積）

複数の要素がすべて必要であることを表します。順序を持たない一組のデータです。

```text
data 注文 = 注文番号
  AND 顧客
  AND 配送先住所
  AND 請求先住所
  AND List<注文明細>
  AND 金額総計
```

要素を別の `data` として分解できます。「住所」や「顧客」のような概念はプリミティブな値まで分解することで、初めて具体的な意味と制約が定まります。

## OR（直和）

複数の場合のうち、ちょうど1つを選択することを表します。**排他**かつ**網羅**です。

```text
data 連絡手段 = メールアドレス OR 電話番号 OR 住所
```

排他: 同時に複数の場合に該当することはありません。網羅: このモデルが扱う範囲では、列挙された以外の場合は存在しません。

`OR` の主な使いどころは次の3つです。

### 状態の分岐（時間軸の差）

ライフサイクルの状態を分けます。状態を経るたびに必須項目が変わる業務概念には、状態ごとに別の `data` を作って `OR` で結合します。

```text
data 注文 = 未承認注文 OR 承認済み注文 OR 出荷済み注文

data 未承認注文 = 注文番号 AND 顧客 AND 金額
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時
data 出荷済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時 AND 出荷日時
```

各状態で必須となる項目が型レベルで保証されます。「承認済みなのに承認者が無い」状態は型として作れません。

### 分類の分岐（静的判定の結果）

同じ時点・同じ状態のデータを、判定結果によって型に刻みます。後続の `behavior` の入力型として使うことで、誤った経路への流入を型で防ぎます。

```text
data 事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請

behavior 事前承認が必要か判断する =
  申請済み出張申請 -> 事前承認必要な出張申請 OR 事前承認不要な出張申請

behavior 上長が事前承認する =
  事前承認必要な出張申請 AND 承認者 -> 事前承認OK OR 事前承認NG
```

「事前承認が必要かどうか」を boolean ではなく型として刻むことで、`上長が事前承認する` は事前承認が必要な申請にしか呼べないことが型で保証されます。

### 形が違うバリアント

同じ役割を持つが構造が違うものを並べます。

```text
data 注文数量 = 個数 OR キログラム

data 支払い方法 = クレジットカード OR 銀行振込 OR 代金引換 OR コンビニ決済

data クレジットカード = カード番号 AND 有効期限 AND セキュリティコード
data 銀行振込 = 振込先口座 AND 振込期限
```

## 任意（`?`）

ある場合とない場合がある要素には `?` を付けます。

```text
data 顧客 = 氏名 AND メールアドレス? AND 電話番号?
```

ただし、`?` を多用すると業務上ありえない状態（メールも電話も両方 null）まで許容してしまいます。本当に表したいのが「メールか電話のどちらかは必須」なら、`?` ではなく `OR` で分けます。

```text
# 良い
data 顧客 = 氏名 AND 連絡先
data 連絡先 = メールアドレス OR 電話番号 OR メールアドレスと電話番号
data メールアドレスと電話番号 = メールアドレス AND 電話番号
```

```text
# 避けたい
data 顧客 = 氏名 AND メールアドレス? AND 電話番号?
```

Wlaschin の "Making illegal states unrepresentable" の議論です。`?` は「業務上、欠落が正当な情報」だけに使います。

## List

同種の要素が複数ある場合は `List<...>` で表します。

```text
data 注文 = 注文番号 AND 顧客情報 AND List<注文明細>
```

特に指定がなければ0件を許容します。「最低1件は必要」ならコメントで明記します。

```text
data 書籍 = ISBN AND タイトル AND List<著者> // 著者は1人以上
```

「メールも電話も持ってよいが、少なくとも1つは必須」のような要件は、`OR` ではなく `List` ＋件数制約で表します。

```text
data 顧客 = 氏名 AND List<連絡手段> // 連絡手段は1件以上
```

## 不正な状態を表現不可能にする

`AND` と `OR` を業務ルールに合わせて組み合わせ、業務上ありえない状態を構造として作れないようにします。Wlaschin の "Making illegal states unrepresentable" の中心的なアイディアです。

「メールアドレスかメールアドレス＋郵便アドレスかメールアドレス＋郵便アドレス」のような業務ルールがあれば、`?` で吸収するのではなく `OR` で 3 ケースを列挙します。

```text
data ContactInfo =
  | EmailOnly           // メールのみ
  | PostOnly            // 郵便のみ
  | EmailAndPost        // 両方
```

15通りの組み合わせのように見えても、深い洞察への refactoring（"Discovering new concepts"）で、`ContactMethod` のリストのように上位概念にまとめられることが多いです。組み合わせ爆発を感じたら、抽象概念が見えていないサインだと考えます。

## イミュータブルデータモデル

時間とともに変化する業務概念は、状態を更新するのではなくイベントを積み重ねて表現します（イミュータブルデータモデル）。

### リソース／イベントの分類

| 種類       | 性質                                              | 動詞テスト           | 例                       |
| ---------- | ------------------------------------------------- | -------------------- | ------------------------ |
| リソース   | 時間を超えて同一性を保つ。日時属性を持たない      | 「〜する」が不自然   | 顧客、製品、会員         |
| イベント   | ある時点で発生した不変の事実。日時属性を1つ持つ   | 「〜する」が成立する | 注文、承認、出荷、削除   |

「マスタ」「トランザクション」という従来の分類は使いません。日時属性の有無という客観的な基準で分けます。

### イベントの設計

イベントには日時属性を1つだけ持たせます。複数の日時属性を持ちそうになったら、別のイベントに分解します。

```text
# 良い：1イベント1日時
data 注文作成イベント = イベントID AND 顧客 AND 注文内容 AND 作成日時
data 注文確認イベント = イベントID AND 対象注文作成イベントID AND 確認者 AND 確認日時
data 注文出荷イベント = イベントID AND 対象注文作成イベントID AND 配送業者 AND 出荷日時
```

```text
# 避けたい：1レコードに複数日時
data 注文 = 注文ID AND 顧客 AND 注文内容
  AND 注文日時 AND 注文確認日時? AND 出荷日時? AND 配達完了日時?
```

### リソースの隠れたイベントを抽出する

リソースに「更新日時」を持たせたくなったら、そこに隠されたイベントがあるサインです。リソースを更新する業務（情報変更、強制退会、復会など）をイベントとして洗い出します。

```text
data 会員 = 会員ID AND 現在の氏名 AND 現在のメールアドレス AND 会員ステータス

data 会員情報変更イベント = イベントID AND 会員ID AND 変更者 AND 新氏名? AND 新メールアドレス? AND 変更日時
data 強制退会イベント = イベントID AND 会員ID AND 実行者 AND 退会理由 AND 退会日時
data 復会イベント = イベントID AND 会員ID AND 実行者 AND 復会理由 AND 復会日時
```

### イベント系列と現在状態の導出

ロングタームイベント（注文、配送、契約など）はイベント系列として表現し、現在状態は `behavior` で導出します。

```text
data 注文イベント系列 = List<注文イベント>
data 注文イベント = 注文作成イベント OR 注文確認イベント OR 注文承認イベント OR 注文出荷イベント

behavior 注文の現在状態を求める = 注文イベント系列 -> 注文の現在状態 OR 状態不整合エラー
```

## 関連のモデリング

エンティティ間の関連は、リソース／イベントの分類で設計判断が決まります（Scrapbox「関連のモデリング」を参照）。

| 関連の種類                          | 設計                                               |
| ----------------------------------- | -------------------------------------------------- |
| リソース×リソース（依存関係あり）   | 親が子の集合を持つ。実体参照、Eager Load           |
| リソース×リソース（依存関係なし）   | 交差エンティティを別途作る。ID参照                 |
| イベント×リソース                   | イベント側がリソースの参照を持つ（実体参照）       |
| イベントの明細                      | イベントが明細の集合を持つ                         |
| イベント×イベント                   | 後発イベントが先行イベントのID参照を持つ           |

詳細は [../../shell/persistence/README.md](../../shell/persistence/README.md) で扱います。Core では関連の業務的な意味を `data` として書き、永続化方式（Eager Load / 差分検出 / DEL/INS など）は Shell に押し出します。

## チェックリスト

`data` を書いたら次を確認します。

- 状態に依存して必須項目が変わるなら、`?` で吸収せず `OR` で状態を分けたか
- フラグやステータスコードで状態を表していないか
- `OR` のケースは排他かつ網羅になっているか
- `?` を使った場合、欠落が業務上正当か
- `List` の件数制約（空禁止／上限／重複禁止）を明示したか
- リソースとイベントを区別し、イベントには日時属性を1つだけ持たせたか
- イベントの日時属性が複数になっていないか
- リソースに「更新日時」を持たせていないか（持たせたいなら隠れたイベントを抽出したか）
- 業務上ありえない状態を、構造として作れないようにしたか
- 抽象概念が見えていない「組み合わせ爆発」になっていないか
