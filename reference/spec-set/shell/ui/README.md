# UI（画面・帳票・メール）への変換規約

Core の `data` を、画面・帳票・メールに必要な項目に投影する規約です。プレゼンテーションモデルの考え方に従いつつ、SIer の現場で必要な「表現先固有の項目構造」を Shell 側で扱います。

## プレゼンテーションモデルとは

プレゼンテーションモデルは、UIへのデータ表示を最適化するモデルです。Core のドメインモデルとは別の関心事を持ちます。

- 表示形式の最適化（数値の桁区切り、日時のフォーマット、文字数省略）
- デバイス別のレイアウト（PC、スマホ、タブレット）
- 言語別の表現（i18n）
- 操作の活性／非活性（ロール・状態に応じて）

これらは Core の `data` には含めません。Core が `data 注文 = 注文番号 AND 顧客 AND 金額 AND ...` であるとき、画面では「注文番号は ORD-{yyyy}-{連番4桁} 形式で表示」「金額は #,##0 円」のような編集仕様が要りますが、それは Core の関心事ではなく Shell の関心事です。

## 派生元参照を必須化する

UI の各項目には、Core の `data` のどの属性から派生したかを明示します。Core にない項目を画面側で発明することは禁止します。

```text
# Core
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時

# Shell（UI - 注文詳細画面）
| 項目名     | 派生元                   | 編集仕様                |
| ---------- | ------------------------ | ----------------------- |
| 注文番号   | 承認済み注文.注文番号    | ORD-{yyyy}-{nnnn}形式   |
| 顧客名     | 承認済み注文.顧客.氏名   | 全角50文字超は省略表示  |
| 金額       | 承認済み注文.金額        | #,##0 円                |
| 承認者     | 承認済み注文.承認者.氏名 | 全角20文字超は省略      |
| 承認日時   | 承認済み注文.承認日時    | yyyy/MM/dd HH:mm        |
```

派生元のない項目が出てきたら、それは Core の `data` に追加すべき業務概念です。Shell に閉じた項目発明は、画面駆動・配線プログラミングの再生産になります。

## 画面

### 画面項目グループ

画面は項目グループ単位で構造化します。各グループは表示する内容と業務的な意味を持ちます。

- ヘッダ領域（注文番号、状態、合計金額など、常に見える主要情報）
- 配送先領域、明細領域などの詳細情報
- 操作領域（ボタン、フォーム）

### デバイス別レイアウト

業務概念は1つで、デバイスごとに表示項目と編集仕様を分けます。

```text
# Core
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 配送先 AND List<明細>

# Shell（UI - 注文詳細画面）

## PC版
すべての項目を表示。明細は表で展開。

## スマホ版
ヘッダ領域は最重要項目（注文番号、状態、金額）のみ。
配送先と明細は折りたたみ表示。タップで展開。
操作ボタンは画面下部に固定。
```

レイアウトバリアントを画面ファイルに並べます。Core からの派生元が同じであることを、各バリアントで確認します。

### 画面イベント

画面上の操作（ボタンクリック、フォーム送信、項目変更など）は、対応する Core の `behavior` を呼び出すか、別画面に遷移します。

```text
| イベント名 | 発生タイミング   | 対応する behavior        | 遷移先              |
| ---------- | ---------------- | ------------------------ | ------------------- |
| onApprove  | 承認ボタン押下   | behavior 注文を承認する  | 承認完了画面        |
| onCancel   | キャンセル押下   | behavior 注文を取消す    | キャンセル完了画面  |
```

### アクセス制御

画面のアクセス制御は、Core で定義したロールに従います。

```text
ロール: SALES_VIEWER 以上で閲覧可
       SALES_MANAGER 以上で承認・キャンセル可
表示: ロールに応じて操作ボタンを活性／非活性
```

「ボタンが押せるか」は画面の関心事ですが、「誰が承認できるか」は Core の関心事です。Core の `behavior` が `マネージャー以上の承認者` を入力に要求すれば、API は不正な呼び出しをエラーで返し、画面はそれに合わせてボタンを非活性にします。

## 帳票

### 物理仕様

帳票は紙に印刷される（または PDF として配布される）成果物なので、物理的な制約があります。

- 用紙サイズ（A3、A4、A5、B4、B5）、用紙向き（縦／横）
- ページ数（固定 / 可変、最大ページ数）
- 出力タイミング（オンライン即時 / バッチ / 月次月初など）
- 出力契機（業務イベント / 定期実行 / ユーザ操作）

これらは Core の関心事ではなく Shell の関心事です。

### 項目グループと出力条件

帳票は項目グループ単位で構造化します。各グループは出力条件を持ちます。

```text
# 帳票: 注文一覧
| 項目グループ | 出力条件                | 2ページ目以降 |
| ------------ | ----------------------- | ------------- |
| 表紙         | 常に出力                | なし          |
| 集計サマリ   | 注文件数 >= 10          | あり          |
| 明細一覧     | 常に出力                | あり          |
| 注釈         | 高額注文が含まれるとき  | あり          |
```

各項目には Core からの派生元と編集仕様を持たせます。揃え（左／中央／右）、フォント（種類／サイズ）、表示最大バイト数なども Shell の項目として記述します。

## メール

### 件名と本文テンプレート

メールテンプレートは、件名と本文に Core の `data` から派生する埋め込み変数を含みます。

```text
件名: 【ご注文承認のお知らせ】注文番号 {0} を承認しました

本文:
{1} 様

下記のご注文を承認いたしました。

ご注文番号: {0}
ご注文日時: {2}
承認日時:   {3}
ご注文金額: {4} 円

...
```

### 埋め込み変数の派生元

各埋め込み変数は Core の `data` の属性から派生し、編集仕様を持ちます。

```text
| プレースホルダ | 派生元                 | 編集仕様                       |
| -------------- | ---------------------- | ------------------------------ |
| {0}            | 承認済み注文.注文番号  | ORD-{yyyy}-{連番4桁}形式に編集 |
| {1}            | 承認済み注文.顧客.氏名 | そのまま                       |
| {2}            | 承認済み注文.注文日時  | yyyy年MM月dd日 HH:mm           |
| {3}            | 承認済み注文.承認日時  | yyyy年MM月dd日 HH:mm           |
| {4}            | 承認済み注文.金額      | #,##0                          |
```

### メールヘッダと多言語対応

差出人（From）、返信先（Reply-To）、宛先（To/Cc/Bcc）、文字コード、メール形式（テキスト / HTML）、添付ファイル、署名は Shell の関心事として記述します。

多言語対応が必要なら、言語ごとにテンプレートを別ファイルにします。

```text
ja: mails/MAIL-001-approve-notification.ja.tmpl
en: mails/MAIL-001-approve-notification.en.tmpl
```

## ナビゲーション

画面遷移は Core の `behavior` の連鎖、または Shell に閉じた画面間の移動として表現します。

```text
# Mermaid で画面遷移を表現
flowchart LR
  Top[トップ画面] --> OrderList[注文一覧]
  OrderList --> OrderDetail[注文詳細]
  OrderDetail -->|承認| ApproveResult[承認完了]
  OrderDetail -->|キャンセル| CancelResult[キャンセル完了]
```

業務上の意味のある遷移（承認、キャンセル）は対応する `behavior` を持ちますが、技術的な遷移（戻る、エラー画面、認証エラー画面）は Shell に閉じます。

## ユーザー入力の検証

ユーザー入力は Shell の境界で Core の `data` 型に変換します。「バリデーションと不変条件の区別」（外部入力の検証 vs ドメインオブジェクトの不変保持）に従います。

- **バリデーション**：外部入力が正しい形式かをチェックする（Shell の責任）
- **不変条件**：ドメインオブジェクトが常に満たすべき制約（Core の型として保証）

画面でのバリデーション（必須チェック、桁チェック、フォーマットチェック）は、ユーザー体験のために即時フィードバックを返す目的で Shell 側にも置きますが、最終的な検証は Core の `behavior` で行います。Core 側で検証を済ませれば、画面以外（API直叩き、バッチ処理）からの入力でも同じ業務ルールが適用されます。
