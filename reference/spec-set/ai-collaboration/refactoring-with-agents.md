# 仕様モデルの洗練にエージェントを使う

SMDD ではエージェントを「Core から実装を生成する」場面で論じることが多いですが、その一段手前 ──仕様モデルそのものを洗練する場面── にもエージェントは活用できます。SIer の現場で多い「既存設計書（画面・テーブル・処理詳細）から仕様モデルを抽出する」プロセスを、エージェントの設計サポートで進めます。

## 前提：外側が先でもインサイドアウトは可能

SMDD のインサイドアウト開発はゼロから Core を書き始める前提ですが、SIer 現場では既存設計書が先にあるケースが大半です。「外側が先に決まる」ことと「インサイドアウトで進める」ことは両立します。既存設計書を素材として、そこから Core を抽出するプロセスがインサイドアウトの実体です。

## 既存設計書からの Core 抽出

### ステップ1：素材の準備

抽出元になる素材を集めます。

- 画面項目定義（Excel、画面遷移図、画面イベント一覧）
- テーブル定義（CREATE TABLE 文、ER図、ドメイン定義書）
- 処理詳細（システム機能設計書の自然言語記述）
- 業務マニュアル、業務ルール集

これらを Markdown / プレーンテキスト化し、エージェントに渡せる形にします。Excel のままだとエージェントが読めないので、`traditional-design-schema.dsl` のような構造化 DSL に変換しておくと良いです。

### ステップ2：5W1Hでエンティティを抽出

5W1H の手順に従い、エージェントに業務記述からエンティティを洗い出させます。

```text
プロンプト例:
次の業務記述からエンティティを 5W1H で抽出してください。
- Who（誰が）：行動する主体、関与する人
- What（何を）：扱う対象、成果物
- When（いつ）：時間に関する概念
- Where（どこで）：場所や位置
- Why（なぜ）：目的や理由を表す行為
- How（どのように）：手段や方法を示す概念

[業務記述をここに貼り付け]
```

エージェントが返したエンティティ候補を、人間が業務知識で確定します。

### ステップ3：リソース／イベントへの分類

動詞テスト（「〜する」が成立するか）で、エンティティをリソースとイベントに分けます。

```text
プロンプト例:
次のエンティティ候補を、リソースとイベントに分類してください。

判断基準:
- リソース: 時間を超えて同一性を保つ。動詞「〜する」が不自然
- イベント: ある時点で発生した事実。動詞「〜する」が成立し、日時属性を持つ

[エンティティリスト]
```

### ステップ4：状態を OR で分ける

エンティティが取りうる状態を `OR` で列挙します。フラグやステータスコードで状態を表している既存設計書からは、状態を構造に持ち上げる作業が必要です。

ここで重要なのは、`OR` の分割軸は通常**複数の候補があり、どの軸を採るかで以降の仕様モデルの形が決まる**ことです。AI が最初に提示した分割をそのまま採用すると、別の軸で切ったほうが業務に合っていた可能性に気づけないまま spec-model が固まります。`Bill = InQuota OR WithOverage` (枠消費の有無で分ける) と `Bill = MonthlyBill OR OneTimeBill` (請求周期で分ける) は、同じ「請求」という対象に対して**直交する別の軸**であり、どちらを採るかは業務ルールと変更履歴に照らして判断する必要があります。

このステップは A1-A3 の3小ステップに分解します。

#### 4-A1：軸候補の列挙

AI に分割候補を**最低3つ**提示させます。1つしか出てこない場合は、別の視点を促すプロンプトを追加します。

```text
プロンプト例:
次のテーブル定義 / 業務記述を見て、エンティティ X の状態を OR で分ける軸を
最低3つ提示してください。それぞれの候補について、

(a) どの不変条件が枝ごとに異なるか
(b) どの業務語彙の対立を反映しているか（例: 「枠内 vs 超過」「月次 vs 単発」）
(c) 業務ルールや変更履歴の中で、その軸に沿って独立に変わった事例があるか

を併記してください。1つの軸しか思い浮かばない場合は、その理由も書いてください。

[テーブル定義 / 業務記述]
```

#### 4-A2：評価チェックリストでの確定

返ってきた候補を、次のチェックリストで評価します。disposability の判断表と整合します。

| 候補の特徴 | 採るべき設計 |
| --- | --- |
| ある枝が他の枝の特殊ケース | 共通の上位型にして特殊ケースを派生にする |
| 枝ごとに不変条件が異なる | spec-model の `OR` の枝として分ける |
| 入力形式 (JSON フィールド名など) だけが違う | spec-model では分けず、Shell の decoder 側で吸収 |
| ユースケースの流れが違う | `behavior` を別に分ける (data の OR ではなく behavior の分離) |
| 永続表現 (DBレイアウト) だけが違う | spec-model では分けず、shell/persistence 側で representation_version で吸収 |

この表のうち**「枝ごとに不変条件が異なる」「ユースケースの流れが違う」**に該当する軸が、spec-model に持ち込むべき軸です。それ以外は Shell 側で吸収すべきで、Core に持ち込むと「コピーしない」ことが目的化して意味のない上位型を作ることになります。

#### 4-A3：確定と Why の記録

選んだ軸と、選ばなかった軸の理由を spec-model 本文の Why セクションに必ず残します。

```text
## 設計の意図

採用した分割: data 請求 = 枠内請求 OR 超過込み請求

理由:
- 枝ごとに不変条件が異なる: 枠内請求は超過レートを持たない (空欄ではなく「フィールドが存在しない」が正しい意味)。超過込み請求は超過トークン数と超過レートを必須とする。型レベルで保証したい不変条件が異なるので、OR の枝として分けた。

採らなかった代替軸:
- 請求周期で分ける案 (月次 vs 単発): 業務上、両者は同じ請求ロジックを通り、不変条件も共通だった。集計周期は behavior の入力として渡せば足り、data の枝にする必要はない。
- 顧客タイプで分ける案 (Premium vs Standard): プラン種別は枠の大きさに反映されるが、請求結果の構造そのものは同じ。Premium 専用フィールドを追加する必要が出た時点で再検討する。
```

「なぜ別の分割を採らなかったか」を残す目的は、将来の枝追加・削除判断の材料を残すためです。新しい業務要件で「単発請求は別ロジックが必要になった」と判明したとき、過去の判断記録があれば「そのときは共通だったが状況が変わった」という形で意思決定を更新できます。

#### 一括プロンプト例

A1-A3 を通したプロンプト例:

```text
プロンプト例:
次のテーブル定義を見ると、status カラム / フラグカラムで状態が表現されています。
SMDD の指針に従って data の OR 分割を提案してください。

ただし以下の手順を踏んでください:

1. 分割軸の候補を最低3つ提示する。各候補について、不変条件の違い、業務語彙の対立、変更履歴の独立性を併記する。
2. 採るべき軸を、次の判断表で評価する。
   - ある枝が他の枝の特殊ケース → 上位型にする
   - 枝ごとに不変条件が異なる → spec-model の OR の枝にする
   - 入力形式だけが違う → Shell の decoder 側で吸収する
   - ユースケースの流れが違う → behavior を分ける
   - 永続表現だけが違う → shell/persistence 側で吸収する
3. 採用した軸と採らなかった軸の両方の理由を Why として書く。

[テーブル定義]
```

### ステップ5：behavior の抽出と全域性の確保

業務操作を `behavior` として書き出し、入力・出力の型を絞ります。

```text
プロンプト例:
次の処理詳細から、業務操作を behavior として抽出してください。
全域性の指針（入力データを制限する）に従って、各 behavior が入力型を絞る形にしてください。
失敗ケースは出力型の OR の一枝として明示してください。

[処理詳細]
```

### ステップ6：Why の抽出

判定規則に Why を添えます。既存設計書には Why が書かれていないことが多いので、業務マニュアルやドメイン専門家への質問から補います。エージェントは「なぜそのルールがあるか」を推測できますが、業務上の正しさは人間が確認する必要があります。

```text
プロンプト例:
次の判定規則について、なぜこのルールが存在するかの業務上の理由を
コメントとして添えてください。推測になる場合は、それを明示してください。

[判定規則]
```

## Wlaschin 型の deeper insight への refactoring

Wlaschin の "Discovering new concepts" で示されているように、組み合わせ爆発に見える問題は、深い洞察への refactoring で解消することが多いです。

例: 連絡手段が「メールアドレス」「郵便アドレス」「自宅電話」「会社電話」の4種類あり、「最低1つは必要」という業務ルールがある場合。15通りの組み合わせを `OR` で列挙する必要があるように見えますが、`ContactMethod` という上位概念を導入すれば `List<ContactMethod>` の最低1件として表現できます。

### エージェントへの依頼

```text
プロンプト例:
次の data 定義は、組み合わせが爆発しているように見えます。
Wlaschin の "Discovering new concepts" の流儀で、
上位概念を抽出して refactoring してください。
- 同じ役割を持つ要素はリストにまとめられないか
- 重複する構造は共通の data に切り出せないか

[data 定義]
```

エージェントが返した refactoring 案を、人間が業務知識で評価します。「`ContactMethod` という上位概念は業務的に存在するか」「業務上の意味のある区別が消えていないか」を確認します。

## 仕様モデル洗練のループ

SMDD の洗練指針（凝集度・結合度・全域性）とイミュータブルデータモデルの指針は、エージェントとの設計対話の指針として使えます。

### 1. 凝集度を上げる

エージェントに「単一責任原則」と「Honest & Complete な命名」を指針として与え、複数の責務を持つ `behavior` を分解させます。

### 2. 結合度を下げる

エージェントに「スタンプ結合の排除」「全域性の確保」を指針として与え、`behavior` の入力型を絞らせます。

### 3. 隠れたイベントを抽出する

エージェントに「リソースに更新日時を持たせたくなったら、そこに隠されたイベントがあるサイン」を指針として与え、業務イベントを洗い出させます。

### 4. 状態の表現を書き換える

エージェントに「フラグではなく OR」「ステータスコードではなく型と遷移」を指針として与え、状態の表現を構造に持ち上げさせます。

各ステップで、エージェントの提案を人間が業務知識で評価し、確定したものを Core に反映します。エージェントは設計の選択肢を広げる協業相手で、最終的な業務上の正しさは人間が判断します。

## 検証

抽出した仕様モデルが6条件を満たすかを、エージェント自身に検証させます。

```text
プロンプト例:
次の仕様モデルが、SMDD の6条件を満たしているか確認してください。
1. 仕様レベルで抽象概念を作っているか（手続き抽象・データ抽象）
2. データの不変条件が一意に定まっているか（状態ごとに型を分けているか）
3. 振る舞いが全域性を満たしているか（入力に対して出力が常に定義されているか）
4. 振る舞いがスタンプ結合でないか（使わないデータを引数に入れていないか）
5. 振る舞いが単一責務であるか（Honest & Complete な命名ができているか）
6. 各ルールに「Why」が添えられているか

満たしていない条件があれば、改善案を提示してください。

[仕様モデル]
```

満たしていない条件があれば、ステップに戻って refactoring を繰り返します。
