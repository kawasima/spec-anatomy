# 出張申請（初期モデル）

書籍7.1節の出張申請の初期モデルです。最終形（[business-trip.md](business-trip.md)）と比較するための参考として残しています。実プロジェクトでは最終形を使ってください。

## 設計の意図

このモデルはまだ書籍8.5節の6条件を完全には満たしていません。次の点で改善の余地があります。

- 状態が文字列フラグで表現されているところが残る
- behavior が出張申請全体を受け取っていてスタンプ結合に近い
- Why のコメントが薄い

書籍8.5節の6条件の指針で refactoring すると、[business-trip.md](business-trip.md) の最終形に到達します。

## 基本的な値の型

```text
data 金額 = 整数 // 0以上、円単位
data 従業員ID = 文字列 // 空文字列ではない
data 日付 = 年月日 // ISO8601形式

data 役職 = 管理職 OR 一般社員
data 管理職 = レベル // レベル1-5
data 一般社員 = 単位型

data 費用負担区分 = 自社負担 OR 先方負担

data 従業員 =
  従業員ID
  AND 氏名
  AND 役職
  AND 上長ID
```

## 出張申請の共通項目

```text
data 出張申請共通項目 =
  申請者
  AND 予定費用
  AND 費用負担区分
  AND 出張先
  AND 出張目的
  AND 出張開始日
  AND 出張終了日
```

## 出張申請の状態

書籍7.1節では状態を OR で列挙し、状態ごとに data を分けます。

```text
data 出張申請 =
  申請準備中
  OR 提出済み
  OR 事前承認待ち
  OR 事前承認却下済み
  OR 事前承認済み
  OR 出張完了
  OR 最終承認待ち
  OR 最終承認却下済み
  OR 承認完了

data 申請準備中 = 出張申請共通項目

data 提出済み =
  出張申請共通項目
  AND 提出日時

data 事前承認理由 =
  高額出張
  OR 権限不足
  OR 先方費用負担

data 高額出張 = 基準金額 // 10万円
data 権限不足 = 役職
data 先方費用負担 = 単位型

data 事前承認待ち =
  出張申請共通項目
  AND 提出日時
  AND 事前承認理由リスト
  AND 承認者ID

data 事前承認理由リスト = List<事前承認理由>

data 事前承認却下済み =
  出張申請共通項目
  AND 提出日時
  AND 却下日時
  AND 却下理由
  AND 承認者ID

data 事前承認済み =
  出張申請共通項目
  AND 提出日時
  AND 事前承認日時
  AND 承認者ID

data 出張完了共通項目 =
  出張申請共通項目
  AND 実費用
  AND 提出日時
  AND 完了日時
  AND 出張報告

data 出張完了 =
  出張完了共通項目
  AND 事前承認日時?
  AND 承認者ID?

data 最終承認待ち =
  出張完了共通項目
  AND 最終承認者ID
  AND 事前承認日時?
  AND 事前承認者ID?

data 最終承認却下済み =
  出張完了共通項目
  AND 却下日時
  AND 却下理由
  AND 最終承認者ID

data 承認完了 =
  出張完了共通項目
  AND 承認日時
  AND 最終承認者ID
  AND 精算額
  AND 事前承認日時?
  AND 事前承認者ID?
```

## behavior 定義

```text
behavior 事前承認要否を判定する =
  予定費用 AND 申請者 AND 費用負担区分 -> 事前承認理由リスト

// 理由:
// 1. 予定費用が10万円以上
// 2. 申請者が一般社員
// 3. 費用負担が先方（接待との境界が曖昧になるため）

behavior 精算額を計算する =
  実費用 AND 費用負担区分 -> 金額
// 自社負担の場合: 実費用をそのまま精算
// 先方負担の場合: 0円を精算

behavior 出張申請を提出する =
  申請準備中 -> 提出済み OR 事前承認待ち
// 事前承認理由リストが空なら提出済み
// 事前承認理由リストが空でないなら事前承認待ち

behavior 事前承認する =
  事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー
// 前提条件: 承認者IDが申請者の上長IDと一致すること

behavior 事前承認を却下する =
  事前承認待ち AND 承認者ID AND 却下理由 -> 事前承認却下済み OR 却下エラー
// 前提条件: 承認者IDが申請者の上長IDと一致すること
// 前提条件: 却下理由が空文字列でないこと

behavior 出張を完了する =
  (提出済み OR 事前承認済み) AND 実費用 AND 出張報告 -> 出張完了 OR 完了エラー
// 前提条件: 実費用が0より大きいこと
// 前提条件: 出張報告が空文字列でないこと

behavior 最終承認を依頼する =
  出張完了 -> 最終承認待ち

behavior 最終承認する =
  最終承認待ち AND 承認者ID -> 承認完了 OR 承認エラー
// 前提条件: 承認者IDが申請者の上長IDと一致すること

behavior 最終承認を却下する =
  最終承認待ち AND 承認者ID AND 却下理由 -> 最終承認却下済み OR 却下エラー
// 前提条件: 承認者IDが申請者の上長IDと一致すること
// 前提条件: 却下理由が空文字列でないこと
```

## 最終形との違い

[business-trip.md](business-trip.md) の最終形では、次のような refactoring が行われています。

1. **「事前承認必要／不要」を分類の分岐として型に刻む**: `data 事前承認必要な出張申請 = 申請済み出張申請 AND 事前承認要件` のように、事前承認要件を持つ申請とそうでない申請を別の型に分けることで、`上長が事前承認する` の入力型が `事前承認必要な出張申請` だけに絞られる
2. **承認者を従業員IDではなく従業員エンティティとして扱う**: 承認者IDから承認者エンティティ（社員）への参照に変えることで、業務概念として扱える
3. **「精算額を計算する」を「最終承認する」の責務に統合**: 最終承認時に経理連携することが業務上の責務なので、独立した behavior にせず最終承認の事後条件として扱う
4. **「却下」と「承認」の結果型を統合**: 事前承認OK／事前承認NG として、却下理由を持つ NG と持たない OK を直和型で表現

書籍3.5節の段階的改善のプロセスを経ると、自然に最終形に到達します。
