# 既存設計書から Core を抽出するサンプルプロンプト

SIer 現場で典型的な「画面・テーブル・処理詳細が先にあり、そこから仕様モデルを抽出したい」場面のサンプルプロンプトです。

## 状況

既存の出張申請システムには次の素材があります。

- 画面項目定義（出張申請入力画面、上長承認画面、最終承認画面）
- テーブル定義（business_trips、business_trip_details、employees）
- 処理詳細（システム機能設計書の自然言語）

これらから仕様モデルを抽出します。

## ステップ1: 5W1H でエンティティを抽出

```text
あなたはドメインモデリングのエージェントです。
5W1H の手順に従って、
次の業務記述からエンティティを 5W1H で抽出してください。

判断基準:
- Who（誰が）: 行動する主体、関与する人
- What（何を）: 扱う対象、成果物
- When（いつ）: 時間に関する概念
- Where（どこで）: 場所や位置
- Why（なぜ）: 目的や理由を表す行為
- How（どのように）: 手段や方法を示す概念

業務記述:
---
従業員が出張予定を入力して出張申請を提出する。承認者は申請内容を確認し、予算と
照らし合わせて承認または却下を行う。承認された場合、従業員は出張を実施し、帰社後に
出張報告と経費明細を提出する。経理担当者が内容を確認して精算を完了する。

ただし、予定費用が10万円以上、または申請者が役職なしの場合は事前承認が必要である。
費用負担区分が先方の場合も、金額および役職に関わらず事前承認が必要である。
---

出力形式:
- Who: ...
- What: ...
- When: ...
- Where: ...
- Why: ...
- How: ...
```

## ステップ2: リソース／イベントへの分類

```text
動詞テスト（「〜する」が成立するか）に従って、ステップ1で抽出した次のエンティティを
リソースとイベントに分類してください。

判断基準:
- リソース: 時間を超えて同一性を保つ。動詞「〜する」が不自然
- イベント: ある時点で発生した不変の事実。動詞「〜する」が成立し、日時属性を持つ

エンティティ:
- 従業員、承認者、経理担当者
- 出張予定、申請内容、予算、出張報告、経費明細
- 申請、承認、却下、精算
- 出張申請書、出張報告書、経費明細書

出力形式:
- リソース: [...]
- イベント: [...]

「マスタ」「トランザクション」という分類は使わないでください。
```

## ステップ3: 状態を OR で分ける

```text
次のテーブル定義を見ると、status カラムで状態が表現されています。

テーブル定義:
---
CREATE TABLE business_trips (
  id, applicant_id, status, submitted_at, approver_id, approved_at, ...
  status IN ('draft', 'submitted', 'pre_approval_required',
             'pre_approved', 'pre_rejected', 'completed', 'finalized')
);
---

SMDD の指針に従って、これを「フラグやステータスコードではなく、状態ごとに
data を OR で分ける」形に書き直してください。状態によって必須となる項目を、
各 data に持たせてください。

特に注意:
- status='pre_approved' のとき approver_id と approved_at が必須
- status='pre_rejected' のとき approver_id、approved_at、rejection_reason が必須
- status='draft' のとき submitted_at は NULL（まだ提出されていない）

出力形式:
仕様DSL のコードブロックで `data 出張申請 = ... OR ...` の形式
```

## ステップ4: behavior の抽出と全域性の確保

```text
ステップ3で得た data 定義を前提に、業務操作を behavior として書き出してください。

全域性の指針（入力データを制限する）に従って、各 behavior が入力型を絞る形にしてください。
失敗ケースは出力型の OR の一枝として明示してください。

業務操作:
- 出張申請する（ドラフトの作成）
- 申請する（ドラフト → 申請済み）
- 事前承認が必要か判断する
- 上長が事前承認する
- 出張実績を登録する
- 最終承認する

各 behavior について:
- 入力型を、その操作が呼ばれる業務状態に絞ること
- 出力型を、成功と失敗の OR で表現すること（例外を投げない）
- 前提条件を //コメント として添えること
```

## ステップ5: Why の抽出

```text
ステップ4 で抽出した behavior のうち、判定規則を持つものについて、
業務上の理由を //コメント として添えてください。

例:
behavior 事前承認が必要か判断する =
  申請済み出張申請 -> 事前承認必要な出張申請 OR 事前承認不要な出張申請
// 事前承認要件:
// - 高額出張(予定費用合計 >= 100,000円)
//   理由: <ここを埋める>
// - 役職なし(申請者の役職 = なし)
//   理由: <ここを埋める>
// - 先方負担(費用負担区分 = 先方負担)
//   理由: <ここを埋める>

業務マニュアルや既存のシステム設計書から推測できる理由を埋め、推測である場合は
「推測:」と明示してください。確証が必要な箇所は「ドメインエキスパートへの確認が必要」
と明示してください。
```

## ステップ6: 6条件の検証

```text
ここまでで抽出した次の仕様モデルが、SMDD の6条件を満たしているか確認してください。

[ステップ3〜5 で抽出した仕様モデル]

6条件:
1. 仕様レベルで抽象概念を作っているか（手続き抽象・データ抽象）
2. データの不変条件が一意に定まっているか（状態ごとに型を分けているか）
3. 振る舞いが全域性を満たしているか（入力に対して出力が常に定義されているか）
4. 振る舞いがスタンプ結合でないか（使わないデータを引数に入れていないか）
5. 振る舞いが単一責務であるか（Honest & Complete な命名ができているか）
6. 各ルールに「Why」が添えられているか

満たしていない条件があれば、改善案を提示してください。
```

## 期待される最終出力

ステップ1〜6を通じて、最終形（[../spec-model/business-trip.md](../spec-model/business-trip.md)）に近い仕様モデルが得られます。エージェントの出力をそのまま採用するのではなく、人間が業務知識で確認しながら確定します。

特に Wlaschin 流の deeper insight への refactoring（"Discovering new concepts"）を意識すると、初回の抽出では見えなかった上位概念（例: `事前承認要件` を独立した型として刻む）が出てきます。エージェントに「上位概念を抽出して refactoring してください」と依頼する追加ステップを設けると効果的です。
