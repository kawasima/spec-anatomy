# 出張申請詳細画面

[../../spec-model/business-trip.md](../../spec-model/business-trip.md) の `事前承認必要な出張申請` を表示し、上長が承認・却下を行う画面の例です。

## 画面の業務的役割

販売部門マネージャー以上が、事前承認必要な出張申請を確認し、承認または却下する。注文一覧から行クリックで遷移してくる。承認操作で `behavior 上長が事前承認する` を呼び出します。

## 関連する behavior

- [上長が事前承認する](../../spec-model/business-trip.md) (Core の `behavior 上長が事前承認する`)

## レイアウトバリアント

### PC版

#### ヘッダ領域

| 項目名       | 派生元                                | 編集仕様                          |
| ------------ | ------------------------------------- | --------------------------------- |
| 申請ID       | 事前承認必要な出張申請.申請済み.id    | BT-{yyyy}-{連番4桁}形式           |
| 申請日時     | 事前承認必要な出張申請.申請済み.申請日時 | yyyy/MM/dd HH:mm                  |
| 申請者       | 事前承認必要な出張申請.申請済み.申請者.氏名 | 全角50文字超は省略表示           |
| 出張目的     | 事前承認必要な出張申請.出張予定.出張目的 | そのまま、長文は省略             |
| 出張期間     | 事前承認必要な出張申請.出張予定.出張期間 | yyyy/MM/dd 〜 yyyy/MM/dd          |
| 予定費用合計 | sum(出張予定.出張予定費用.予定金額)    | #,##0 円                          |
| 費用負担     | 事前承認必要な出張申請.出張予定.費用負担区分 | 「自社負担」または「先方負担」 |

#### 事前承認要件領域

事前承認要件を箇条書きで表示します。

```text
データ:
事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請

表示:
□ 高額出張（予定費用合計が10万円以上）
□ 役職なし申請（申請者の役職が「役職なし」）
□ 先方負担申請（費用負担区分が「先方負担」）

該当する要件にチェックを表示。理由ツールチップを表示できる。
```

#### 予定費用明細領域

| 項目名     | 派生元                          | 編集仕様       |
| ---------- | ------------------------------- | -------------- |
| 出張予定日付 | 出張予定日程.出張予定日付       | yyyy/MM/dd     |
| 費目       | 出張予定日程.費目               | 「交通費」「宿泊費」「交際費」 |
| 予定金額   | 出張予定日程.予定金額           | #,##0 円       |

予定費用は List<出張予定日程> なので、行ごとに展開して一覧表示します。

#### 操作領域

| 項目名         | 種別   | 編集仕様       | 表示条件                                    |
| -------------- | ------ | -------------- | ------------------------------------------- |
| 承認ボタン     | button | ラベル「承認」 | role >= MANAGER かつ 承認者が申請者の上長   |
| 却下ボタン     | button | ラベル「却下」 | role >= MANAGER かつ 承認者が申請者の上長   |
| 却下理由入力   | textarea | 1〜500文字     | 却下ボタン押下時にモーダルで表示            |
| 戻るボタン     | button | ラベル「一覧に戻る」 | 常に表示                              |

### スマホ版

スマホは画面幅の制約で、ヘッダ領域は最重要項目のみ、明細は折りたたみ表示にします。操作ボタンは画面下部に固定配置します。

#### ヘッダ領域（スマホ）

| 項目名       | 派生元                              | 編集仕様             |
| ------------ | ----------------------------------- | -------------------- |
| 申請ID       | 事前承認必要な出張申請.申請済み.id  | BT-{yyyymmdd}-{連番4桁}（短縮形）|
| 出張目的     | 事前承認必要な出張申請.出張予定.出張目的 | 全角20文字超は省略 |
| 予定費用合計 | sum(出張予定.出張予定費用.予定金額) | #,##0 円             |

#### 事前承認要件領域（スマホ）

PC版と同じ。

#### 予定費用明細領域（スマホ）

折りたたみ表示。タップで展開。1明細1行のカード表示。

#### 操作領域（スマホ）

画面下部に固定。承認・却下ボタンを横並び。

## 画面イベント

| イベント名 | 発生タイミング      | サーバ通信 | 対応する API           | 正常時遷移先     |
| ---------- | ------------------- | ---------- | ---------------------- | ---------------- |
| onApprove  | 承認ボタン onClick  | あり_同期  | POST /business-trips/{id}/pre-approve（decision=approve） | 承認完了画面 |
| onReject   | 却下ボタン onClick + 却下理由入力後 | あり_同期 | POST /business-trips/{id}/pre-approve（decision=reject） | 却下完了画面 |
| onBack     | 戻るボタン onClick  | なし       | -                      | 出張申請一覧画面 |

### onApprove の詳細

API レスポンスに応じた遷移を Shell で扱います。

- 200 PreApproved → 承認完了画面へ遷移
- 200 PreApprovalRejected → このパターンは onReject 用なので想定外（エラー画面）
- 409 InvariantViolation → 「他の操作によって状態が変わりました。再読み込みします」とトーストし画面再読み込み
- 403 AuthorizationError → 「承認権限がありません」とエラー表示

## アクセス制御

- 認証必須: はい
- 必要ロール: SALES_VIEWER 以上（閲覧）、SALES_MANAGER 以上（操作）
- 動的な制御: 操作ボタンの活性／非活性を表示条件で制御（上の表参照）

## 派生元参照の規約

このSpec Setの規約として、画面の各項目には Core の data からの派生元を明示します。Core にない項目を画面側で発明することは禁止します。

「予定費用合計」のように Core で直接保持していない項目（`sum(出張予定.出張予定費用.予定金額)` の計算結果）は、Core 側に対応する `behavior` を作るか、画面側で派生計算として明示します。今回は後者として「派生元: sum(...)」と書いています。
