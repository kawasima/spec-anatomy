# Shell: 実装モデルへの変換規約

Core（仕様モデル）から、API・永続モデル・UI・メッセージング契約への変換規約を置く場所です。Shell は Core を参照しますが、Core は Shell を知りません（一方向の依存）。

書籍6章（仕様モデルから実装モデルへ）と書籍8.3.2の Phase 2（Shell の実装）に従います。

## Shell の役割

Shell は Core を取り囲む外部インターフェースの設計です。

- ユーザー入力を Core が理解する型に変換する（アダプター）
- Core の出力を UI や DB に適した形に変換する（アダプター）
- 永続化やネットワーク通信のような副作用を扱う

書籍6章で論じられるとおり、実装モデルには3種類があります。

- **ドメインモデル**: ビジネスロジックを表現する（Core そのもの、または Core から派生）
- **永続モデル**: データベースとのマッピングを担う
- **プレゼンテーションモデル**: UIへのデータ表示を担う

3つは扱う関心事が異なるので、別々の型として定義します。1つのクラスに兼ねさせると、画面都合・DB都合・業務ルールが同じ場所で衝突します。

## Core からの派生

Shell の各成果物は、Core を参照する形で定義されます。

### API

Core の `behavior` を HTTP/gRPC のエンドポイントに対応させ、入出力 `data` を OpenAPI/Protobuf スキーマに変換します。

```text
# Core
behavior 注文を承認する = 未承認注文 AND 承認者 -> 承認済み注文 OR 承認エラー

# Shell（API）
POST /api/v1/orders/{orderId}/approve
リクエスト: 承認者 (Core の data 承認者 から派生)
レスポンス（200）: 承認済み注文（Core から派生）
レスポンス（409）: 承認エラー（Core から派生）
```

### 永続モデル

Core の `data` を、テーブル定義に変換します。Active Record・Data Mapper・Repository などのパターンで、Core と永続層を分離します。

```text
# Core
data 注文 = 未承認注文 OR 承認済み注文 OR 出荷済み注文

# Shell（永続）
table orders {
  id, customer_id, amount, status, approver_id (NULL可), approved_at (NULL可)
  # status カラムが Core の OR 分岐を表す
  # NULL可カラムが Core の状態依存項目を表す
}
```

### UI

Core の `data` を、画面・帳票・メールに必要な項目に投影します。表現先固有の関心事（デバイス別レイアウト、項目の編集仕様）は Shell の責任です。

```text
# Core
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時

# Shell（UI - 注文詳細画面 PC版）
| 項目名     | 派生元              | 編集仕様                |
| ---------- | ------------------- | ----------------------- |
| 注文番号   | 承認済み注文.注文番号 | ORD-{yyyy}-{nnnn}形式  |
| 顧客名     | 承認済み注文.顧客.氏名 | 全角50文字超は省略表示 |
| 金額       | 承認済み注文.金額    | #,##0 円               |
```

### メッセージング

Core のドメインイベントを、メッセージブローカーのトピックに対応させます。

```text
# Core
data 注文承認イベント = ...
behavior 注文を承認する = ... -> 承認済み注文 AND List<注文承認イベント>

# Shell（メッセージング）
topic order.approved
ペイロードスキーマ: 注文承認イベント（Core から派生）
配信保証: At-least-once
順序保証: パーティションキー = order_id
```

## Shell が守ること

### Core を変更しない

Shell の都合で Core を変更しません。例えば「テーブル設計の都合で `data` 構造を平坦にしてほしい」「画面の都合で項目を追加してほしい」のような要求があれば、Shell 側で吸収します。

Shell が吸収しきれず Core の変更が必要になった場合は、それは業務ルールの変更です。Core の変更は書籍7章（仕様変更への対処）と書籍8.7（変更を安全に届ける）の流儀で扱います。

### Core からの派生を必須化する

Shell の各項目は Core から派生する派生元参照を持ちます。「画面でだけ使う項目」「帳票でだけ使う項目」を Core に存在しないまま発明することは禁止します。

派生元のない項目が出てきたら、それは Core の `data` に追加すべき業務概念です。Shell に閉じた項目発明は、書籍8.1.3で批判される画面駆動・配線プログラミングの再生産になります。

### 派生のための情報も Core に持つ

「PC版とスマホ版で表示項目が違う」のように、Shell に固有の情報がある場合でも、その判断基準（どの項目が業務上の主要項目か、どの項目が補足項目か）を Core 側で記述しておきます。Core で「重要度」のような区分を `data` として持っていれば、Shell はその区分を参照して画面ごとに表示／非表示を切り替えられます。

Core にない判断を Shell の都度判断にすると、画面ごとに判断が揺れます。

## Shell の変更方針

### Strict Spec ではなく Policy & Discretion

Shell の生成にエージェントを使うときは、書籍8.6 の Policy & Discretion の流儀に従います。

- 言語、フレームワーク、ライブラリの選定は方針として与え、エージェントの裁量に任せる
- データベースの設計、マイグレーション、テンプレートエンジン、ORMの使い方は方針で示し、詳細は委ねる
- ただし「Core の値を変えるな」「Core から派生していない項目を作るな」は Strict Spec として渡す

詳細は [../ai-collaboration/shell-generation.md](../ai-collaboration/shell-generation.md)。

### 変更の経済性

書籍8.4の経済性逆転は Core で成立する話です。Shell（特に UI、永続）はフレームワーク依存・ライブラリ依存が強いため、エージェントによる再生成だけでは不十分なケースも残ります。Shell 側の変更は、Core の変更を起点に、人間の確認とエージェントの提案を交互に行います。

## 性能優先と Core/Shell 分離

「JOIN 1回で取れるものを Core 経由で 2 クエリにするのは遅い」という性能議論で Core/Shell の分離を崩したくなる場面があります。書籍6.2.3 で論じられているとおり、性能議論の多くは「面倒のため」を性能の名で語っているだけのことが多いです。

性能を優先する場合は次の順で考えます。

1. 遅いことが観測されているか、SLA から推定されているかを確認する
2. 遅いのが書き込み側か読み取り側かを特定する
3. 多くの場合は読み取り側の特定クエリだけが遅いので、CQRS 的に読み取り経路だけを永続モデル直結にし、書き込み側の Core/Shell 分離は守る
