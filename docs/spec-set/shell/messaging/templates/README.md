## メッセージングと外部I/Fの違い

両者ともシステム間通信の契約を扱いますが、性格が異なります。

| 項目         | メッセージング              | 外部I/F                        |
| ------------ | --------------------------- | ------------------------------ |
| 通信方式     | 非同期（pub-sub、キュー）   | 同期 HTTP / 非同期ファイル連携 |
| 配信保証     | At-most-once 〜 Exactly-once | 通常はリクエストレスポンス     |
| 順序保証     | パーティションごとに保証可  | リクエスト単位                 |
| ペイロード形式 | JSON / Avro / Protobuf      | JSON / XML / 固定長 / CSV      |
| 用途         | ドメインイベントの伝播      | レガシーシステム連携、バッチ連携 |

## Core との関係

### メッセージング

Core のドメインイベントを外部に通知する場合は、メッセージング契約として記述します。

```text
# Core
data 注文承認イベント = ...
behavior 注文を承認する = ... -> 承認済み注文 AND 注文承認イベント

# Shell（メッセージング）
topic: order.approved
ペイロードスキーマ: 注文承認イベント（Core から派生）
```

### 外部I/F

外部システムとの連携で、Core にない形式（固定長、レガシー XML）を扱う場合は、外部I/F契約として記述します。Core の `data` への変換は境界（パーサ／ジェネレータ）で行います。

```text
# Shell（外部I/F）
ファイル: 注文情報送信.dat（固定長）
電文項目: 注文番号(20桁), 顧客コード(8桁), 金額(10桁)

# 境界での変換
behavior parseOrderFile = 固定長レコード -> Core の data 注文 OR パースエラー
```

## 派生元の書き方

メッセージングのペイロード項目と外部I/Fの電文項目は、Core の `data` から派生します。

```text
| 項目名     | 派生元                              | 編集仕様              |
| ---------- | ----------------------------------- | --------------------- |
| 注文番号   | 注文承認イベント.対象注文ID         | そのまま              |
| 承認者     | 注文承認イベント.承認者.id          | そのまま              |
| 承認日時   | 注文承認イベント.承認日時           | ISO 8601 形式         |
```

派生元のないペイロード項目が出てきたら、それは Core に追加すべき業務概念です。
