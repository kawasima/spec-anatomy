# ID 体系・採番規則

システム全体で使う識別子の体系と採番規則です。Traditional SI 設計書の §11.4 採番一覧に対応します。

## ID 体系一覧

| ID 種別     | 物理名         | フォーマット         | シーケンス名      | 採番方式 | サイクリック | 初期値 | 最小値 | 最大値    | 増分 | 関連する Core の data |
| ----------- | -------------- | -------------------- | ----------------- | -------- | ------------ | ------ | ------ | --------- | ---- | --------------------- |
| 注文ID      | order_id       | `ORD-{yyyy}-{連番4桁}` | seq_order         | DB シーケンス + フォーマット | あり（年度別） | 1 | 1 | 9999 | 1 | `data 注文.id` |
| 顧客ID      | customer_id    | ULID（26文字）       | -                 | アプリ側生成 | -            | -      | -      | -         | -    | `data 顧客.id` |
| 申請ID      | application_id | `APP-{yyyymmdd}-{連番4桁}` | seq_application_daily | DB シーケンス + フォーマット | あり（日次） | 1 | 1 | 9999 | 1 | `data 出張申請.id` |

## 採番方式

| 方式                   | 用途                                            | 利点                            | 欠点                              |
| ---------------------- | ----------------------------------------------- | ------------------------------- | --------------------------------- |
| DB シーケンス          | 連番が業務上必要な ID（注文番号、申請番号）   | 連番の保証、業務担当者が読みやすい | DB のシングルポイント、分散難しい |
| ULID（時系列ソート可能）| 内部 ID（顧客 ID、メッセージ ID）              | 分散環境で衝突しない、時系列ソート可能 | 業務担当者が読みづらい           |
| UUID v4                | グローバルに一意な ID                           | 完全分散                        | 時系列ソート不可                  |
| Snowflake ID 等        | 高頻度の ID 採番                                | 高速、時系列ソート可能          | ノード ID の管理が必要            |

## 採番のタイミング

| ID 種別 | 採番タイミング             | 理由                                       |
| ------- | -------------------------- | ------------------------------------------ |
| 注文ID  | 注文確定時（DB INSERT 時） | 確定前のドラフトには ID を振らない         |
| 申請ID  | 提出時                     | 下書き状態には ID を振らない               |
| 顧客ID  | 顧客登録時                 | -                                          |

## サイクリック採番

サイクリック採番（年度ごと、日次でリセット）の場合は、リセットの境界を明示します。

```text
注文ID: ORD-{yyyy}-{連番4桁}
- 年度: 4月1日〜翌年3月31日
- 4月1日に連番が 0001 にリセットされる
- 9999 を超えたら運用警告（業務的に通常は超えない想定）
```

サイクリック採番の場合、ID だけで一意に識別できないことがあります（例: 同じ連番が異なる年度に発行される）。グローバルな一意性が必要な場面では、別途 UUID または ULID を内部 ID として持ちます。
