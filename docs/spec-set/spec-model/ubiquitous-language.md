# ユビキタス言語

業務ドメインで使う語彙を定義する場所です。仕様DSL で `data` や `behavior` の名前として登場する語と、それらが指す業務的意味を揃えます。

ユビキタス言語は単なる用語集ではなく、`syntax`（語の組み合わせ規則）、`semantics`（語の使われ方）、`pragmatics`（文脈による意味の差）を含む半形式の記述です。Scrapboxの「ユビキタス言語」の整理に従います。

## 構成

実プロジェクトでは、本ファイルに業務領域全体の用語を集約するか、業務領域ごとにファイルを分けるかを選びます。小さな領域なら1ファイルで足り、複数の Bounded Context にまたがるなら `contexts/<context-name>.md` のように分割します。

## 用語の書き方

### 名詞

業務概念としての名詞。`data` の名前として登場することが多い。

```markdown
### 注文 (Order)

顧客が販売部門に対して購入意思を表明したもの。注文ヘッダ（顧客・日付・配送先）と
1つ以上の注文明細から成る。

- 別名: 受注（販売部門の業務文脈）
- 近接概念との違い: 見積依頼とは違う。見積依頼は購入意思を伴わない問い合わせ
- 関連する data: `data 注文` (../data/order.md)
- 出現する Bounded Context: 受注、販売
```

### 動詞

業務操作としての動詞。`behavior` の名前として登場することが多い。

```markdown
### 承認する (approve)

未承認の注文を、業務責任者が確認して受理可能と判断する操作。承認後は注文内容の
変更ができなくなる。

- 主体: 販売部門マネージャー以上
- 入力: 未承認の注文
- 出力: 承認済み注文（承認者・承認日時が記録される）
- 関連する behavior: `behavior 注文を承認する` (../behavior/order-approval.md)
```

### 状態

エンティティのライフサイクル状態としての名詞。状態を `OR` で分けるときの各バリアントとして登場します。

```markdown
### 未承認 (Pending)

顧客が注文を確定し、業務責任者の承認待ちの状態。

- 該当する data: `data 未承認注文` (../data/order.md)
- 入る経路: 顧客が注文確定操作を行った時点
- 出る経路: 「承認する」によって「承認済」へ
- この状態で許される操作: 内容修正、キャンセル、承認
- この状態で禁止される操作: 出荷指示
```

## 用語間の関係

包含・対立・近接・別名のような関係を、Mermaid 等で図示できます。詳細はテキスト側で書きます。

```text
graph LR
  注文 -->|包含| 注文明細
  注文 -->|対立| 見積依頼
  受注 -.別名.-> 注文
```

## 改訂履歴

用語の追加・削除・意味の変更は履歴として残します。コードと並走させる必要があるので、命名変更があったときは関連するコード側のシンボルや PR への参照を書きます。

```markdown
| 日付       | 変更内容                                            | 関連PR        |
| ---------- | --------------------------------------------------- | ------------- |
| 2026-04-25 | 「承認する」のロール条件を追加（運用ルール変更）    | #123          |
```

## ユビキタス言語と仕様DSL の関係

仕様DSL の `data` 名と `behavior` 名は、ユビキタス言語と一致させます。

- ユビキタス言語に書かれていない名詞や動詞を、仕様DSL の中で唐突に使わない
- 仕様DSL でリファクタリングして名前を変えたら、ユビキタス言語も同じタイミングで更新する

このSpec Setでは、命名の食い違いをエージェントが検出する仕組みを [../ai-collaboration/verification-loop.md](../ai-collaboration/verification-loop.md) で扱います。
