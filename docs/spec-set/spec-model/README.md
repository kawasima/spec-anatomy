# Core: 仕様モデル

仕様DSL（`data` と `behavior`、必要に応じて `workflow`）でシステムの仕様を記述する場所です。Core はフレームワーク・DB・UI への依存を一切持ちません。

仕様モデル駆動設計（SMDD）で導入されたドメイン記述ミニ言語の文法に従います。

## 仕様DSLの基本

### 構成要素

- **data**: システムが扱う情報の構造を定義します
- **behavior**: システムが行う処理を、入力と出力の関係として定義します
- **workflow**: 複数の `behavior` を順序づけて連結したもの

### 演算子

- `=` 定義の左辺と右辺を結びます
- `AND` 複数の要素がすべて必要であることを示します
- `OR` 複数の場合のうち、ちょうど1つを選択することを示します（排他かつ網羅）
- `?` 任意（ある場合とない場合がある）を示します
- `List<...>` 同種の要素が複数あることを示します。特に指定がなければ0件を許容します
- `->` 振る舞いの入力から出力への変換を示します
- `//` コメント

### 例

```text
data 注文 = 未承認注文 OR 承認済み注文

data 未承認注文 = 注文番号 AND 顧客 AND 金額
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時

behavior 注文を承認する = 未承認注文 AND 承認者 -> 承認済み注文 OR 承認エラー
```

## ファイル構成の規約

### 1ファイル1業務領域

関連する `data` と `behavior` をまとめた業務領域単位（注文ドメイン、出張申請ドメインなど）で1ファイルにします。1つの `data` ごとにファイルを分けると、関連する状態と遷移が散らばって読みにくくなります。

### Front matter

各ファイルの冒頭には最小限の Front matter を置きます。

```yaml
---
name: <業務領域の名前>
description: <1行の説明>
status: draft / approved / deprecated
last-reviewed: YYYY-MM-DD
---
```

`axis` や `kind` のような分類タグ、`related.aggregates` のような関連参照は置きません。仕様DSL のコードブロックの中で参照する型名と、本文中の Markdown リンクで関連を表現します。

### 本文

本文は仕様DSL のコードブロックと、各定義の意図・理由（Why）を散文で書きます。コードブロックには言語指定として `text` または `domain` を使います。

```text
# 注文ドメイン

## data 定義

\```text
data 注文 = 未承認注文 OR 承認済み注文 OR キャンセル済み注文
...
\```

## 設計の意図

なぜこの状態分割を採ったか、なぜこの不変条件が必要かを散文で書きます。
Why はAIへの生成指示として機能するので、必ず書きます。

## behavior 定義

\```text
behavior 注文を承認する = 未承認注文 AND 承認者 -> 承認済み注文 OR 承認エラー
\```
```

## 仕様モデルの書き方の柱

### 状態は OR で分ける

「フラグ」や「ステータスコード」で状態を表すのは仕様隠しです。状態ごとに `data` を別の型として分け、その状態でのみ意味を持つ項目を AND で合成します。

```text
# 良い：状態ごとに型を分ける
data 注文 = 未承認注文 OR 承認済み注文

data 未承認注文 = 注文番号 AND 金額
data 承認済み注文 = 注文番号 AND 金額 AND 承認者 AND 承認日時
```

```text
# 避けたい：フラグで状態を持つ
data 注文 = 注文番号 AND 金額 AND 承認済みフラグ AND 承認者? AND 承認日時?
```

詳しくは [data/README.md](data/README.md) と Wlaschin の "Designing with types: Making illegal states unrepresentable" を参照してください。

### 振る舞いは全域性を持たせる

`behavior` は入力 `data` のすべてに対して、結果（成功または失敗）が `data` として定義されている状態を保ちます。例外を投げて済ませる、`null` を返す、未定義の入力を許す、のいずれもしません。失敗も `data` の一枝として明示します。

```text
# 良い：失敗を data として表現
behavior 注文を承認する = 未承認注文 AND 承認者 -> 承認済み注文 OR 承認エラー
```

全域性を達成する手段は3つあります。

1. デフォルト値を返す
2. 出力データにエラーケースを含める
3. 入力データを制限する（推奨）

仕様モデルの段階では3を第一に検討します。`ゼロでない整数` のように、振る舞いが安全に扱える型を入力に要求することで、エラーケースをデータ型の境界に押し込みます。

### 不正な状態を表現不可能にする

`OR` と `AND` を業務ルールに合わせて組み合わせ、業務上ありえない状態を構造として作れないようにします。「メールアドレスか電話番号のどちらかは必須」を `?` で表すと「両方とも null」が許せてしまうので、`OR` で `EmailOnly OR PhoneOnly OR EmailAndPhone` のように書きます。Wlaschin の "Designing with types: Making illegal states unrepresentable" の議論です。

### Why を必ず添える

判定規則と data の境界には、それが業務上どんな意味を持つかをコメントで添えます。Why をプロンプトに含めるかどうかでエージェントが生成するコードの形が変わります。Why のない `cost >= 100000` は条件式の羅列に落ちますが、`高額出張` と名前を付ければエージェントは型として表現します。

```text
data 事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請

// 理由:
// - 高額出張: 高額支出は組織として事前に妥当性を確認する必要がある
// - 役職なし申請: 役職なしの社員は支出判断の権限を持たないため上長の判断が必要
// - 先方負担申請: 接待との境界が曖昧になるため不適切な支出を防ぐ事前チェックが必要
```

### スタンプ結合を避ける

`behavior` は使わないデータまで含む大きな型を引数に取らないようにします。実際に使う最小限のデータだけを受け取る型に絞ります（スタンプ結合の排除）。

```text
# 良い：必要な型だけ受け取る
data 出張期間 = 開始日 AND 終了日
behavior 出張期間を検証する = 出張期間 -> 検証済み期間 OR 期間矛盾エラー
```

```text
# 避けたい：使わないフィールドまで受け取る
behavior 出張期間を検証する = 出張申請 -> 検証済み期間 OR 期間矛盾エラー
```

## イミュータブルデータモデル

業務上の事実が時間とともに変化する場合は、状態を更新する設計ではなく、イベントの積み重ねとして表現します（イミュータブルデータモデル）。

- リソース（顧客・製品など、時間を超えて同一性を保つもの）には日時属性を持たせない
- イベント（注文・承認・出荷など、ある時点で発生した事実）には日時属性を1つだけ持たせる
- リソースの変化もイベントとして記録する
- 関連の設計はイベント／リソースの分類で決める（Scrapboxの「関連のモデリング」を参照）
