# 構造の正しさ：data の不変条件テスト

`data` に付随する制約を不変条件として捉え、それをテストで固定する規約です。書籍5.4節に従います。

## 不変条件の3種類

### 値域制約

単一の値が取りうる範囲。

- `金額` は0以上
- `使用人数` は1以上
- `文字列` は空でない、最大長がある

### 相互制約

複数の値の間に成立すべき関係。`AND` で合成したデータで頻出します。

- `出張開始日 ≤ 出張終了日`
- `予定費用の合計 = 明細の合計`

### 集合制約

`List` を使ったときに発生する制約。

- 空リスト禁止（1件以上）
- 上限（最大100件など）
- 重複禁止（同じキーが2回出ない）
- 一貫性（合計金額が明細の合算と一致）

## 型で潰せる制約はテスト以前に弾く

型（AND/OR/`?`/`List`）で表現できる制約は、不正な状態を作れない構造に寄せます。テストはその上で残った制約を対象にします。

```text
# 型で潰せる：状態ごとに型を分ける
data 注文 = 未承認注文 OR 承認済み注文
data 承認済み注文 = 注文番号 AND 顧客 AND 金額 AND 承認者 AND 承認日時
# 「承認済みなのに承認者が無い」は型として作れない
```

```text
# 型で潰せない：値域、相互制約、集合制約
data 出張期間 = 出張開始日 AND 出張終了日
# 「開始 ≤ 終了」は型では表現しづらい
```

## AND 由来の不変条件テスト

### 日付の前後関係

```text
data 出張期間 = 出張開始日 AND 出張終了日
# 不変条件: 開始 ≤ 終了
```

テスト観点:

- 正常系: 開始 = 終了、開始 < 終了 を通す
- 異常系: 開始 > 終了 を拒否し、`日付矛盾エラー` のような明示的なエラー型を返す

異常系は例外で落とすのではなく、仕様モデル上のエラー（`OR` の一枝）として扱います。

### 合計と内訳の整合

```text
data 注文 = 注文番号 AND 金額総計 AND List<注文明細>
# 不変条件: 金額総計 = sum(明細.金額)
```

テスト観点:

- 仕組みのテスト: 合算ロジックが正しいか
- 仕様のテスト: 内訳のパターンが網羅されているか

## OR 由来の不変条件テスト

### 状態の必須差分

`OR` で状態を分けていれば、必須項目の違いは型で保証されます。テストで確認すべきは「分割が業務として適切か」です。

- 業務上重要な区別が落ちていないか
- 後から足す可能性が高い区別を、いま無理に `OR` にしていないか

### ケースの網羅性

`OR` の枝が業務上のすべての場合を網羅しているかを確認します。新しい場合が出てきたら、`OR` の追加が必要です。

## Optional（`?`）のテスト観点

`?` を使った場合、「無いケース」が業務として正当か、`behavior` がそのケースをどう扱うかを必ずテストで固定します。

- 「無い」場合に `behavior` がどう振る舞うべきか（エラーか、デフォルトか）
- 「無い」状態が他の制約と矛盾しないか

`?` を入れた時点で「無いケース」の仕様が必ず発生します。仕様テストはそれを「無いケースを仕様として扱う」ための強制力になります。

## List の制約テスト

```text
data 出張申請 = ... AND List<費用明細> // 1件以上、上限50件
```

テスト観点:

- 空リスト（0件）は拒否される
- 1件は通る
- 上限+1件（51件）は拒否される
- 重複がある場合の扱い（拒否／正規化）

境界値が仕様モデルから導けるようにします。導けないなら仕様が未確定ということです。

## モデル品質チェック

不変条件をテストする前に、モデルの「匂い」を早期に検出するチェックを行います。

- `data` 名が「情報」「データ」「管理」で終わっていないか
- `?` が多すぎないか（`OR` で状態分割すべきではないか）
- `OR` のケースが「増える前提」なのに閉じていないか
- `List` に「ルール」（空禁止・上限・整合）が付いているか

匂いを早期に検出すれば、テストの量を増やすより前に、テストしやすい形にモデルを整えられます。

## 性質ベーステストでの記述例

```text
property "注文金額は明細小計の合計と等しい" {
  forall (order: 検証済み注文) {
    order.金額総計 == sum(order.明細.金額)
  }
}

property "出張期間は開始 ≤ 終了" {
  forall (period: 検証済み出張期間) {
    period.出張開始日 <= period.出張終了日
  }
}
```
