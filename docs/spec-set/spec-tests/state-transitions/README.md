# 時系列の正しさ：状態遷移とイベント系列のテスト

複数の `behavior` を順序づけて適用したとき、状態遷移とイベント系列の整合が保たれるかを検証する規約です。書籍5.6節に従います。

## 状態×操作の組み合わせ表

`behavior` が状態を遷移させる場合、状態×操作の組み合わせ表を作って網羅性を確認します。

```text
状態:      申請準備中、事前承認待ち、事前承認済み、事前承認却下済み、提出済み
操作:      提出する、事前承認する、却下する
組み合わせ: 5 × 3 = 15通り
```

各組み合わせに対して、結果は次の3つのいずれかになります。

- 遷移する（次の状態が定義されている）
- エラーを返す（状態不整合エラーが定義されている）
- 未定義（仕様が欠けている）

仕様テストとして確認すべきは、「未定義」が存在しないことです。すべての状態×操作の組み合わせに対して、遷移先またはエラーが `data` として定義されていれば、状態遷移の全域性が確保されています。

|                   | 提出する                       | 事前承認する                | 却下する                      |
| ----------------- | ------------------------------ | --------------------------- | ----------------------------- |
| 申請準備中        | 提出済み / 事前承認待ち        | 状態不整合エラー            | 状態不整合エラー              |
| 事前承認待ち      | 状態不整合エラー               | 事前承認済み / 承認エラー   | 事前承認却下済み / 却下エラー |
| 事前承認済み      | 状態不整合エラー               | 状態不整合エラー            | 状態不整合エラー              |
| 事前承認却下済み  | 状態不整合エラー               | 状態不整合エラー            | 状態不整合エラー              |
| 提出済み          | 状態不整合エラー               | 状態不整合エラー            | 状態不整合エラー              |

空セルがあれば、仕様が未定義です。この表が埋まっていることが、状態遷移の全域性テストの合格基準です。

## 正当な系列のテスト

正当な系列とは、仕様モデル上で許可された遷移だけを辿る操作の列です。

```text
申請準備中 →[提出する]→ 事前承認待ち →[事前承認する]→ 事前承認済み
```

正当な系列のテストでは、次を確認します。

- 系列の各ステップで、状態が期待通りに遷移するか
- 系列の最終状態が業務上の完了条件を満たすか
- 系列を通じて `data` の不変条件が維持されるか

## 不正な系列のテスト

不正な系列（禁止された操作を含む系列）のテストでは、次を確認します。

- 禁止された操作が適用された時点で、状態不整合エラーが返るか
- エラー後に状態が変化していないか（操作が「何もしなかった」ことの保証）

```text
申請準備中 →[事前承認する]→ 状態不整合エラー（状態は申請準備中のまま）
```

エラー後の状態保証は重要です。不正な操作で状態が中途半端に変わってしまうと、以降の操作すべてが信頼できなくなります。「エラーなら状態は変わらない」を事後条件として定義し、テストで固定します。

## イベント系列からの現在状態の導出

書籍4章のイミュータブルデータモデルで、イベント系列から現在状態を導出する場合は、性質ベーステストで次の性質を検証します。

### 性質1: 正当な系列なら有効な状態を返す

```text
property "正当なイベント系列から有効な状態が導出される" {
  forall (events: 正当な注文イベント系列) {
    fold(events) is 注文の現在状態
  }
}
```

### 性質2: 不正な系列なら状態不整合エラー

```text
property "承認の前に提出がなければ状態不整合" {
  fold([事前承認イベント]) is 状態不整合エラー
}
```

### 性質3: スナップショットからの再開が全体と一致する

```text
property "途中までの fold + 残りの fold は全体の fold と一致する" {
  forall (events: 正当な注文イベント系列, n: 整数) {
    let prefix = events[0..n]
    let suffix = events[n..]
    fold(events) == fold(fold(prefix), suffix)
  }
}
```

これが成り立てば、4章のイベント導出モデルが状態遷移と整合している証拠です。成り立たない場合、イベントの設計と状態遷移の設計に矛盾があります。

## 禁止遷移と禁止順序

「やってはいけないこと」の検証は見落としやすい領域です。正当な遷移は業務フローとして自然に定義されますが、禁止遷移は「書かない限り存在しない」ためです。

### 単一ステップの禁止

組み合わせ表で「状態不整合エラー」と定義されたセル。

- 提出済みの申請に「提出する」を適用 → 状態不整合エラー
- 事前承認済みの申請に「事前承認する」を適用 → 状態不整合エラー

これらは組み合わせ表の全域性で検出できます。表を埋める作業自体が、単一ステップの禁止を網羅する行為です。

### 系列の禁止（職務分離など）

単一ステップでは合法でも、特定の順序で実行すると業務上許されない系列。

- 同じ人物が申請と承認の両方を行う（職務分離の制約）

このような条件は状態遷移だけでは表現できません。次のいずれかで扱います。

- `behavior` のガード条件に系列条件を含める（例: 「承認者 ≠ 申請者」）
- 横断不変条件として定義する（複数の `data` や `behavior` をまたぐ制約）

## 横断不変条件

複数の `data` や `behavior` にまたがる制約。

```text
# 横断不変条件: 予算消化額の合計が予算総額を超えない
property "全申請の予算消化額の合計 <= 予算総額" {
  forall (approvals: List<事前承認済み>) {
    sum(approvals.予定費用) <= 予算総額
  }
}
```

横断不変条件は、個々の `behavior` のテストでは検出できません。複数の状態を経由したあとの整合性を、性質として検証します。
