# 仕様テスト

仕様モデル（`data` と `behavior`）が矛盾なく、抜けなく、変更に耐える形で記述できているかを検証するテストの規約です。実装テストとは目的が異なります。

## 仕様テストと実装テストの違い

- **実装テスト**：実装モデルが期待通りに動くことを確認する
- **仕様テスト**：仕様モデルが矛盾なく、抜けなく、変更に耐える形で記述できていることを確認する

仕様テストの入力と出力は仕様モデル上の `data` であり、HTTPやSQLやUIは登場しません。実装技術の変更（フレームワーク移行、DB変更、UI変更）に対する仕様テストへの影響は最小化されます。一方、業務ルール自体が変われば仕様テストは壊れるべきで、それは正しい振る舞いです。

## 3つの軸

仕様テストの対象は3つの軸に分解されます。

- **構造の正しさ**：`data` の不変条件（[invariants/](invariants/)）
- **変換の正しさ**：`behavior` の全域性（[totality/](totality/)）
- **時系列の正しさ**：状態遷移とイベント系列の整合（[state-transitions/](state-transitions/)）

3つは独立ではなく重なり合います。`behavior` の事後条件テストは `data` の不変条件テストと重なり、状態遷移テストは `behavior` の全域性テストと重なります。軸を分けておくことで「いま書いているテストはどの観点を検証しているか」を明確にします。

## 構造の正しさ：data の不変条件

`data` が「常に成り立つべき条件」を満たしているかを検証します。具体的には次のような不変条件です。

- **値域制約**：金額が0以上、使用人数が1以上、文字列が空でない
- **相互制約**：開始日 ≤ 終了日、合計 = 内訳合算、承認済みなら承認日時が存在する
- **集合制約**：空リスト禁止、上限件数、重複禁止、同一性

型で潰せる制約（`OR` で状態を分ければ自動的に必須になる項目など）は、テスト以前に作れない構造に寄せます。型で潰せない制約（値域、相互制約、集合制約）はテスト対象です。

詳細は [invariants/README.md](invariants/README.md)。

## 変換の正しさ：behavior の全域性

`behavior` が定義された入力すべてに対して結果を返すかを検証します。

- すべての入力区分に対して、成功または失敗のどちらかが定義されているか
- 失敗が `data` として表現されているか（例外や未定義に漏れていないか）
- ガード条件を満たさない入力の扱いが仕様に含まれているか
- 成功時の事後条件（生成物の必須性、不変条件の維持）が満たされるか

詳細は [totality/README.md](totality/README.md)。

## 時系列の正しさ：状態遷移とイベント系列

複数の `behavior` を順序づけて適用したとき、状態遷移とイベント系列の整合が保たれるかを検証します。

- 各状態から許可される操作と禁止される操作が網羅されているか
- 禁止される操作を適用したときに状態不整合エラーが返るか
- 操作の系列を通じて不変条件が維持されるか
- イベント系列から現在状態を導出した結果が一貫しているか

詳細は [state-transitions/README.md](state-transitions/README.md)。

## 型の区分は同値クラスになる

SMDD の型の粒度設計の判断フローは、テスト設計と直結します。仕様モデルで型として定義した区分は、そのままテストの同値クラスとして使えます。

```text
data 金額区分 = 三千円未満 OR 三千円以上
data 地域区分 = 離島 OR 離島以外
```

この定義から、同値クラスは「金額区分 × 地域区分」の4組み合わせとして自動的に決まります。テストケースは代表値と境界値で構成します。型の粒度設計を済ませておけば、「どこで同値クラスを切るか」という最も迷いやすい問いには答えが出ています。

## テスト粒度

3つの軸のそれぞれで、次の粒度を選びます。

1. **ルール単体**：1つの不変条件、または1つの `behavior` を単独で検証する。最も小さく、最も書きやすく、原因特定がしやすい。ほとんどの仕様テストはこの粒度です
2. **ワークフロー**：複数の `behavior` を順序づけて適用し、一連の操作が整合的に完了するかを検証する
3. **横断不変条件**：複数の `data` や `behavior` にまたがる制約を検証する

実際の設計では、まずルール単体のテストを充実させ、その上でワークフローや横断不変条件のテストを必要に応じて追加します。

## 性質ベーステストとの相性

`data` の不変条件、`behavior` の全域性、イベント系列から現在状態への射影は、性質ベーステスト（property-based testing）と特に相性がよい領域です。Haskell の QuickCheck、Java の jqwik、Python の Hypothesis、Scala の ScalaCheck などが利用できます。

ただしツールは後から選べます。重要なのは、性質が仕様モデル側で明文化されていることです。

## Gherkin との相補的関係

Gherkin（Given/When/Then）は仕様モデルの**外延的表現**として、人間が仕様の正しさを直感的に理解・検証するために使えます。一方、仕様モデルは**内包的表現**として、構造と制約を曖昧さなく定義します。

このSpec Setの推奨は次の通りです。

1. 仕様モデルで構造と制約を定義する（内包）
2. 仕様モデルから Gherkin を生成して人間レビューに使う（外延）
3. Gherkin を手で維持するのではなく、仕様モデルを唯一の真実の源泉とする

SMDD では仕様DSL から Gherkin を生成する変換ルールが定義されています。

## 仕様変更とテスト破壊

仕様モデルへの変更はテストへの影響で3種類に分かれます。

- **拡張**: ORの枝を追加する、Listの要素型を増やすなど。既存テストは壊れず、新しいテストが必要
- **制約強化**: 値域を狭める、Optionalを必須にするなど。既存の「通るテスト」の一部が壊れるべき
- **破壊的変更**: data の構造を組み替える、behavior の入出力型を変えるなど。広範囲のテストが壊れる

「拡張なのにテストが壊れた」「制約強化なのにテストが壊れない」「破壊的変更の影響範囲が想定外に広い」は、いずれもテスト設計や仕様モデルの見直しのサインです。
