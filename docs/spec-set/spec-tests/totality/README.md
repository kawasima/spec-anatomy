# 変換の正しさ：behavior の全域性テスト

`behavior` が定義された入力すべてに対して結果を返すか、失敗が `data` として表現されているかを検証する規約です。

## 全域性のテスト

`behavior` の全域性とは、入力 `data` のすべてに対して、結果（成功または失敗）が `data` として定義されている状態を指します。

全域性を欠く `behavior` は次の形で現れます。

- ある入力に対して「何が起きるか分からない」（未定義）
- ある入力に対して「例外が投げられる」（定義の外に漏れている）
- ある入力に対して「実装者が判断する」（仕様が委譲されている）

仕様モデルの立場では、これらは仕様の欠陥です。

## テストの構造

1. `behavior` の入力型（`data` のOR分岐を含む）から、入力の区分を列挙する
2. 各区分に対して、出力型のどの枝に該当するかを確認する
3. どの枝にも該当しない入力区分があれば、仕様が未定義である

```text
behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー

# テスト対象の区分:
# - 入力1: 事前承認待ち AND 上長IDと一致する承認者ID → 事前承認済み に進む
# - 入力2: 事前承認待ち AND 上長IDと一致しない承認者ID → 承認エラー に進む
```

すべての入力区分に対して、出力型のどこに着地するかが定義されていれば、全域性が確保されています。

## 失敗のエラー型設計

失敗を `data` として設計するときの分類は、次の3種類で整理します。

### 検証エラー

入力 `data` が不変条件を満たさない場合のエラー。`data` の不変条件と直結します。

```text
behavior 注文金額を検証する = 注文金額 -> 検証済み注文金額 OR 注文金額検証エラー
# 不変条件: 金額が0以上、必須項目が空でない
```

### 権限エラー

操作する主体が、その振る舞いを適用する資格を持たない場合のエラー。

```text
behavior 上長が事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー
# 承認エラー: 承認者IDが申請者の上長IDと一致しない
```

### 状態不整合エラー

現在の状態が、その振る舞いの前提状態と合致しない場合のエラー。

```text
# 「事前承認待ち」を入力に要求することで、提出済み・承認済みからの呼び出しは型で防ぐ
behavior 事前承認する = 事前承認待ち -> 事前承認済み OR 承認エラー
```

### エラー型のテスト観点

- **区別**: 異なる原因のエラーが、異なるエラー型として区別されているか
- **十分性**: `behavior` の出力型（OR）に、想定されるエラーがすべて含まれているか

エラーを「失敗」と一括りにせず、なぜ失敗したかが仕様から読み取れる粒度を保ちます。

## ガード条件のテスト

ガード条件は2段階で設計します。

### 1. ガードの列挙

`behavior` の定義から、適用条件を明示的に列挙します。

```text
behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認エラー
# ガード条件: 承認者IDが申請者の上長IDと一致すること
```

### 2. ガード違反時の出力を検証

ガード条件を満たさない入力に対して、出力が定義されているかを確認します。

- ガード条件を満たす → 成功枝に進む可能性がある
- ガード条件を満たさない → エラー枝に進むことが定義されている

ガード条件を「呼び出し側の責任」として仕様の外に出さず、ガード違反時の振る舞いも仕様に含めます。

### 複数のガード条件

1つの `behavior` に複数のガード条件がある場合、次を確認します。

- 各ガード条件を個別に違反させたとき、対応するエラーが返るか
- 複数のガード条件を同時に違反させたとき、どのエラーが優先されるか

「権限エラーと検証エラーが同時に発生する場合、どちらを返すか」を仕様として決めておかないと、実装者ごとに判断が分かれます。

## 事後条件のテスト

`behavior` が成功したときに保証される条件を検証します。

### 生成物の必須性

成功時の出力 `data` が、定義されたすべてのフィールドを持っているか。

```text
data 事前承認済み = 申請内容 AND 承認者 AND 承認日時
# 事後条件: 承認者と承認日時が必ず設定されている
```

### 不変条件の維持

`behavior` の実行後も、`data` の不変条件が維持されているか。

- 明細を追加した後も、合計金額 = 明細合算 が成り立つか
- 状態を遷移した後も、許可された状態の範囲内にあるか

### 再実行時の一貫性

再実行時に同じ結果を返すか、状態不整合エラーを返すかを仕様として決めます。

```text
# 冪等な振る舞い: 同じ結果を返す
# 1回目: 成功（事前承認済み）
# 2回目: 同じ結果（事前承認済み）

# 非冪等な振る舞い: 状態不整合エラーを返す
# 1回目: 成功（事前承認済み）
# 2回目: 状態不整合エラー
```

どちらが正しいかは業務次第ですが、仕様として決まっていなければテストも書けません。

## 副作用を含む場合の扱い

`behavior` が副作用を持つ場合は、副作用を出力 `data` の一部として表現します。

```text
behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認結果

data 事前承認結果 = 事前承認済み AND 通知対象リスト OR 承認エラー
```

仕様テストは「`通知対象リスト` が正しく生成されるか」を検証し、「実際にメールが送られるか」は実装テスト（境界）に委ねます。

## 性質ベーステストでの記述例

```text
property "事前承認は事前承認待ちからしか呼べない" {
  forall (request: 事前承認待ち, approver_id: 承認者ID) {
    let result = 事前承認する(request, approver_id)
    result is 事前承認済み or result is 承認エラー
  }
}

property "承認者IDが上長IDと一致しなければ承認エラー" {
  forall (request: 事前承認待ち, approver_id: 承認者ID) {
    if approver_id != request.申請者.上長ID then
      事前承認する(request, approver_id) is 承認エラー
  }
}
```
