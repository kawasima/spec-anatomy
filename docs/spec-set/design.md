# Spec Set 設計方針

このSpec Setの設計判断とその根拠を記述します。実装する側は [README.md](README.md) と [spec-model/README.md](spec-model/README.md) から読み始めてください。本文書は「なぜこの構造にしたか」を説明します。

## 中心に置くものは仕様DSL

仕様モデル駆動設計が答えようとしている問いは「コーディングエージェントに何を渡せば、業務として正しいコードが返ってくるか」です。書籍8.6 の整理に従えば、エージェントには次の2種類を渡します。

- 本質的複雑さ（業務ルール）を **Strict Spec** として、曖昧さゼロで渡す
- 偶有的複雑さ（実装技術）を **Policy & Discretion** として、方針と裁量で渡す

Strict Spec の中身が仕様DSL（`data` と `behavior`）です。自然言語の仕様書、画面仕様書、テーブル定義のいずれもこの役割を果たせません。エージェントが「もっともらしく補完」する余地を残してしまうからです。

このSpec Setは、エージェントへの入力として機能する仕様DSL を Core に置き、そこから派生する API・永続・UI・メッセージングを Shell として位置づけます。

## なぜ集約中心の構造を採らないか

書籍1.3.2節と8章で繰り返し批判されているように、DDD の戦術的パターン（Entity・Value Object・Aggregate・Repository）の機械的適用は、本書が扱う仕様モデル駆動とは別の方向に向かいがちです。集約は仕様モデルを書いた結果として現れる一表現にすぎず、業務概念モデルの中心単位ではありません。

このSpec Setでは、`aggregate-model.md` のような集約中心のテンプレートを置きません。代わりに、`spec-model/data/` と `spec-model/behavior/` のディレクトリに仕様DSL のコードブロックを書き、状態を `OR` で分け、業務ルールを `behavior` のシグネチャに刻みます。

## なぜステータスコード型を含めないか

`OrderStatus = Pending OR Approved OR Shipped OR ...` のような enum を独立した型として定義する設計は、書籍2.2節で「仕様隠し」として批判されています。状態に応じて必須項目が変わる業務ルールが、コードの分岐に埋もれてしまうためです。

仕様モデル駆動の流儀では、状態ごとに data そのものを分けます。`data 注文 = 未承認注文 OR 承認済み注文` のように、状態を data の構造として持ち込み、各状態で必須となる項目を type レベルで保証します。`OrderStatus` のような enum を Core に置くと、状態と項目の対応関係が消えるため、このSpec Setでは Shell（永続モデル）の規約として扱います。

## なぜ表現先固有のテンプレートを Core に置かないか

画面・帳票・メール・電文の項目構造は、Traditional SI設計書では網羅的に定義されてきました。各表現先には固有の関心事（画面なら種別・領域・イベント、帳票なら揃え・フォント、メールなら埋め込み文字列、電文ならバイト長・開始位置・多重度）があり、これらは長年の運用で実装に必要十分な項目セットになっています。

ただしこれらを Core に置くと、書籍8.1.1で批判される「テーブル駆動・画面駆動」と同じ構造になります。画面項目もテーブルカラムも一時点のスナップショットでしかなく、業務概念の時間軸の構造（状態遷移、ライフサイクル）を持ちません。

このSpec Setでは、Traditional SI設計書の項目構造を **Shell の規約** として位置づけ直します（[shell/ui/](shell/ui/)）。各項目は Core の data から派生する派生元参照と編集仕様を必須化することで、エージェントが Core に存在しない項目を画面側で発明するのを防ぎます。

## なぜ Front matter を使わないか

旧版のSpec Setでは、各ファイルの冒頭に `axis`／`kind`／`related.aggregates` のような Front matter を必須化していました。これはエージェントと検証ツールが軸種別と関連を機械的に取れるようにするためでしたが、書籍を読み返すと、この設計はエージェントが本来読むべきもの（仕様DSL）と別の場所にメタデータを持つ構造になっており、エージェントへの入力としての機能を弱めていました。

新版では、各ファイルは仕様DSL のコードブロックを中心に書きます。Front matter は冒頭の `name` と `description` 程度の最小限に留め、関連は本文中の Markdown リンクと、仕様DSL の中で参照する型名で表します。

## なぜ AI Collaboration を正規構成要素として置くか

書籍8章は「Core からのコード生成にエージェントを使う」を中心に論じていますが、SIer の現場ではその一段手前──仕様モデル自体の洗練──にもエージェントが活用できます。

たとえば既存設計書（画面・テーブル・処理詳細）から仕様モデルを抽出する作業は、書籍2章〜4章の指針（凝集度・結合度・全域性・OR で状態を分ける・隠れたイベントを探す）を踏まえれば人間に可能ですが、コーディングエージェントを設計サポートとして使えば、より広範囲の素材を一気に読み、洗練の候補を提案できます。Wlaschin の "Discovering new concepts" のような **deeper insight への refactoring** をエージェントが提案し、人間が確定する形のループは、Spec Set の運用規約として明示的に置く価値があります。

このSpec Setは [ai-collaboration/](ai-collaboration/) を正規構成要素として置き、次の4つを規約化します。

- `refactoring-with-agents.md`：仕様モデル洗練のエージェント活用（既存設計書からの Core 抽出、deeper insight への refactoring の駆動）
- `core-generation.md`：書籍8.6 の Strict Spec をエージェントに守らせる規約
- `shell-generation.md`：書籍8.6 の Policy & Discretion を Shell 生成で活用する規約
- `verification-loop.md`：生成のたびに型・テスト・lint で確認する検証ループの規約

## SIer 現場でインサイドアウトを始める経路

書籍8.3はゼロから Core を書き始める例を示しますが、SIer の現場では既存設計書（画面・テーブル・帳票）が先にあるケースが大半です。このSpec Setは次の経路を支援します。

1. 既存の画面項目・テーブルカラム・処理詳細を素材として、コーディングエージェントに仕様モデルへの抽出を依頼する
2. エージェントが返した仕様モデル案に対し、書籍3章・4章の洗練（凝集度・結合度・全域性・OR で状態を分ける・隠れたイベントを探す）をエージェントが提案し、人間が確定する形で繰り返す
3. Wlaschin の deeper insight への refactoring を、エージェントの設計提案として駆動する
4. Core が固まったら、書籍8.3.2の Phase 2 として Shell（API・UI・永続）をエージェントに生成させる

「外側が先に決まる」ことと「インサイドアウトで進める」ことは両立します。

## 仕様モデルが満たすべき6条件（書籍8.5）

このSpec Setに置く仕様モデルは、書籍8.5節の6条件を満たすものとします。

1. 仕様レベルで抽象概念を作る（手続き抽象・データ抽象）
2. データの不変条件が一意に定まる（状態ごとに型を分ける）
3. 振る舞いが全域性を満たす（入力に対して出力が常に定義される）
4. 振る舞いがスタンプ結合でない（使わないデータを引数に入れない）
5. 振る舞いが単一責務である（Honest & Complete な命名ができる）
6. 各ルールに「Why」が添えられている（理由が判定規則と一緒に書かれている）

これらは書籍8.1.3 のニューレガシーアンチパターン（画面駆動・テーブル駆動・配線プログラミング・儀礼的レイヤリング・モデルオーバーローディング・仕様隠し・Contract-less Service）への処方箋として働きます。

## 検証

このSpec Setが機能しているかは、次で検証します。

- `spec-model/` の中身を生成AIに渡したとき、書籍8.6 のプロンプト例と整合する形で実装コードが生成できるか
- 生成された実装が、`spec-tests/` の規約に従ったテスト（property-based test、状態遷移の網羅、契約テスト）で検証可能か
- `examples/business-trip/` の仕様モデルが、書籍7.1節と8.5.7節の出張申請仕様と整合しているか
- 書籍8.5の6条件を、Spec Set 上の任意の仕様モデルファイルが満たすか
