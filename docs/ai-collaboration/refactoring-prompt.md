# Refactoring プロンプト（出張申請）

SIer 現場で典型的な **「画面・テーブル・処理詳細が先にあり、そこから仕様モデルを抽出したい」** 場面のサンプルプロンプトです。エージェントを設計サポートとして使い、段階的に Core を抽出・洗練させます。

規約は [../../reference/spec-set/ai-collaboration/refactoring-with-agents.md](../../reference/spec-set/ai-collaboration/refactoring-with-agents.md) を参照。

## このプロンプトを使う場面

- 既存システムの設計書（画面、テーブル、処理詳細）から仕様モデルを抽出したい
- 現状の仕様モデルが OR で状態を分けていない、boolean フラグで状態を表しているなど SMDD の6条件を満たしていない
- Wlaschin 流の「deeper insight への refactoring」を実施したい

## 状況（出張申請の例）

既存の出張申請システムには次の素材があります：

- 画面項目定義（出張申請入力画面、上長承認画面、最終承認画面）
- テーブル定義（business_trips、business_trip_planned_costs、business_trip_actual_costs）
- 処理詳細（システム機能設計書の自然言語）

これらから仕様モデルを抽出し、6条件を満たす最終形まで refactoring します。

## ステップ 1: 5W1H でエンティティを抽出

````text
あなたはドメインモデリングのエージェントです。
5W1H の手順に従って、次の業務記述からエンティティを抽出してください。

判断基準:
- Who（誰が）: 行動する主体、関与する人
- What（何を）: 扱う対象、成果物
- When（いつ）: 時間に関する概念
- Where（どこで）: 場所や位置
- Why（なぜ）: 目的や理由を表す行為
- How（どのように）: 手段や方法を示す概念

業務記述:
---
従業員が出張予定を入力して出張申請を提出する。承認者は申請内容を確認し、予算と
照らし合わせて承認または却下を行う。承認された場合、従業員は出張を実施し、帰社後に
出張報告と経費明細を提出する。経理担当者が内容を確認して精算を完了する。

ただし、予定費用が10万円以上、または申請者が役職なしの場合は事前承認が必要である。
費用負担区分が先方の場合も、金額および役職に関わらず事前承認が必要である。
---

出力形式:
- Who: ...
- What: ...
- When: ...
- Where: ...
- Why: ...
- How: ...
````

## ステップ 2: リソース／イベントへの分類

````text
動詞テスト（「〜する」が成立するか）に従って、ステップ1で抽出した次のエンティティを
リソースとイベントに分類してください。

判断基準:
- リソース: 時間を超えて同一性を保つ。動詞「〜する」が不自然
- イベント: ある時点で発生した不変の事実。動詞「〜する」が成立し、日時属性を持つ

エンティティ:
- 従業員、承認者、経理担当者
- 出張予定、申請内容、予算、出張報告、経費明細
- 申請、承認、却下、精算
- 出張申請書、出張報告書、経費明細書

出力形式:
- リソース: [...]
- イベント: [...]

「マスタ」「トランザクション」という分類は使わないでください。
業務的な意味で分類してください。
````

## ステップ 3: 状態を OR で分ける

````text
次のテーブル定義を見ると、status カラムで状態が表現されています。

テーブル定義:
---
CREATE TABLE business_trips (
  id, applicant_id, status, submitted_at, approver_id, approved_at, ...
  status IN ('draft', 'submitted', 'pre_approval_required',
             'pre_approved', 'pre_rejected', 'completed', 'finalized')
);
---

SMDD の指針に従って、これを「フラグやステータスコードではなく、状態ごとに
data を OR で分ける」形に書き直してください。状態によって必須となる項目を、
各 data に持たせてください。

特に注意:
- status='pre_approved' のとき approver_id と approved_at が必須
- status='pre_rejected' のとき approver_id、approved_at、rejection_reason が必須
- status='draft' のとき submitted_at は NULL（まだ提出されていない）

出力形式: 仕様DSL のコードブロックで `data 出張申請 = ... OR ...` の形式
````

## ステップ 4: behavior の抽出と全域性の確保

````text
ステップ3で得た data 定義を前提に、業務操作を behavior として書き出してください。

全域性の指針（入力データを制限する）に従って、各 behavior が入力型を絞る形にしてください。
失敗ケースは出力型の OR の一枝として明示してください（例外を投げない）。

業務操作:
- 出張申請する（ドラフトの作成）
- 申請する（ドラフト → 申請済み）
- 事前承認が必要か判断する
- 上長が事前承認する
- 出張実績を登録する
- 最終承認する

各 behavior について:
- 入力型を、その操作が呼ばれる業務状態に絞ること
- 出力型を、成功と失敗の OR で表現すること（例外を投げない）
- 前提条件を //コメント として添えること
- 仕様モデルが「ありえない遷移」を型で防ぐ形になっていること
````

## ステップ 5: Why の抽出

````text
ステップ4 で抽出した behavior のうち、判定規則を持つものについて、
業務上の理由を //コメント として添えてください。

例:
behavior 事前承認が必要か判断する =
  申請済み出張申請 -> 事前承認必要な出張申請 OR 事前承認不要な出張申請
// 事前承認要件:
// - 高額出張(予定費用合計 >= 100,000円)
//   理由: <ここを埋める>
// - 役職なし(申請者の役職 = なし)
//   理由: <ここを埋める>
// - 先方負担(費用負担区分 = 先方負担)
//   理由: <ここを埋める>

業務マニュアルや既存のシステム設計書から推測できる理由を埋め、推測である場合は
「推測:」と明示してください。確証が必要な箇所は「ドメインエキスパートへの確認が必要」
と明示してください。
````

## ステップ 6: deeper insight への refactoring

これがエージェントを設計サポートとして使う最も価値のあるステップです。Wlaschin 流の "Discovering new concepts" を実施します。

````text
ここまでで抽出した仕様モデルを見直し、より深い業務概念が隠れていないか
refactoring してください。

具体的に探すパターン:
1. **boolean フラグや enum で表現された業務概念**
   例: `is_high_amount: bool, is_no_role: bool, is_counterparty: bool`
   → これらが組み合わさって「事前承認が必要」という1つの判断になっているなら、
     `事前承認要件 = 高額出張 OR 役職なし申請 OR 先方負担申請` の List として上位概念化

2. **複数の if 条件が同じ目的を共有している**
   例: 高額・役職なし・先方負担の3条件で if が分岐
   → 「事前承認要件」という業務概念を導入

3. **状態名がフェーズと一致しているが、フェーズで必須項目が変わる**
   例: `承認者?: 社員, 承認日時?: Date` という Optional
   → `事前承認必要な出張申請` と `事前承認OK` を別の型に分ける

4. **2つ以上の状態に共通する要素がある**
   例: 「事前承認OK」と「最終承認」の両方に「承認者」がある
   → 共通部分を `data 承認イベント = 承認者 AND 承認日時` として抽出するか検討

提案された上位概念に対して:
- 業務的に意味のある名前を付ける
- 仕様DSL で書き直して提示する
- なぜこの refactoring が良いかを Why として説明する

提案を採用するかは人間が確定します。エージェントは複数の選択肢を出してください。
````

## ステップ 7: 6条件の検証

````text
ここまでで抽出した次の仕様モデルが、SMDD の6条件を満たしているか確認してください。

[ステップ3〜6 で抽出した仕様モデル]

6条件:
1. 仕様レベルで抽象概念を作っているか（手続き抽象・データ抽象）
2. データの不変条件が一意に定まっているか（状態ごとに型を分けているか）
3. 振る舞いが全域性を満たしているか（入力に対して出力が常に定義されているか）
4. 振る舞いがスタンプ結合でないか（使わないデータを引数に入れていないか）
5. 振る舞いが単一責務であるか（Honest & Complete な命名ができているか）
6. 各ルールに「Why」が添えられているか

各条件について:
- 満たしているなら「○」と理由
- 満たしていないなら「×」と改善案

改善案は仕様DSL で具体的に書いてください。
````

## ステップ 8: 仕様テスト性質の抽出

````text
最終形の仕様モデルから、仕様テストの3軸（構造／変換／時系列）で検証すべき性質を
列挙してください。

1. 構造の正しさ（不変条件）
   - data が常に満たすべき性質
   - 例: 出張予定費用合計は 0 以上、出張開始日 <= 出張終了日

2. 変換の正しさ（全域性）
   - 各 behavior が入力に対して必ず出力型のいずれかに着地する
   - 同値クラス分割（境界値）

3. 時系列の正しさ（状態遷移）
   - golden path
   - 分岐path
   - 禁止遷移（型レベルで防がれること）

各性質について、property-based test の述語として書ける形で出力してください。
（例: `forall 申請: 申請済み出張申請. 予定費用合計(申請.出張予定) >= 0`）
````

## エージェント駆動 refactoring の運用

このリポジトリには上記ステップを経て得られた最終形が配置されています：

- [../spec-model/business-trip.md](../spec-model/business-trip.md) — 最終形
- [../spec-model/business-trip-initial.md](../spec-model/business-trip-initial.md) — 初期形

`-initial.md` は素朴に抽出した最初の版で、`business-trip.md` がステップ6（deeper insight refactoring）を経た最終形です。差分を見ると、エージェント駆動 refactoring で何が起きるかがわかります。

## 注意

- エージェントの出力をそのまま採用しない。**人間が業務知識で確認**して確定する
- 推測した Why は **ドメインエキスパートに確認** する
- 6条件チェックで「×」が残ったまま Core 生成に進まない（Core 生成時に問題が顕在化する）
- ステップ6 (deeper insight) は1回で終わらない。Core 生成 → 仕様テスト → 違和感 → 仕様モデルに戻る、を繰り返す
